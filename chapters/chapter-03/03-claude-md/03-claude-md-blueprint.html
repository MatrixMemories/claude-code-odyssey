Chapter 3 · Claude Code Full Stack Section 3.3 · The CLAUDE.md file

CLAUDE.md Blueprint

Practical sections, rules & examples that actually deliver

The CLAUDE.md file gives your AI coding partner perfect, persistent context about your project. It turns every fresh session into a seamless continuation instead of a guessing game.

This practical guide walks you through the battle-tested what/why/how framework and a flexible section-based template (with 16 ready-to-use sections, clear include/avoid rules, and real-world examples) so you can build a concise, human-readable CLAUDE.md that dramatically improves Claude’s accuracy, speed, and alignment with your team’s standards.

Subsection 3/8 ~17 min read Claude Code ≥ 2.1

The what/why/how framework is designed as the core foundation providing the three clear, high-level dimensions for building CLAUDE.md files. In practice, one of the most popular and battle-tested templates used by the community is a section-based structure, with anywhere between 4 to 10 sections. In the remainder are some common sections that can be found across many projects, including advice on what to include and what to avoid as well as a practical example for every section.

Note that there is no officially required format for the CLAUDE.md file. It is important to keep it concise, short and human-readable. The section-based template helps keeping the instructions short and concise while excelling at human-readability as well, and is therefore a community favorite. Despite the section-based template not using what/why/how as the top-level headings, the content inside each section is usually designed to cover all three layers.

💡 TIP
Format matters: Concise bullet-point instructions are 40% more likely to be followed than long paragraphs.

1. Project Overview (Essential)

The "Project Overview" section should include a one-liner description of the project that is used for orientation at the beginning of each session. Claude Code has no memory between sessions and thus will not remember anything about the project from previous interactions. Providing a sharp project description will prevent Claude Code from misidentifying the project type, choosing the wrong patterns, or building features for the wrong audience.

The project overview is ideally positioned at the beginning of the CLAUDE.md file to ensure it is the first thing that Claude Code reads. It answers: what is this project, what does it do, and why does it exist?

✅ Include
  • State the product name and core purpose in one sentence
  • Identify the primary technology or service powering the backend
  • Describe the intended user type (developers, researchers, end users)
  • Note the deployment target or runtime environment if it shapes decisions (local, Docker, cloud)
  • Important business or UX constraints
❌ Avoid
  • Don't write more than 3–4 sentences — orientation, not a README file
  • Don't list features — describe what the project is, not what it does in detail
  • Don't include aspirational goals, roadmap items, or marketing language
  • Don't repeat what is obvious from the folder structure
  • Don't use generic statements like “we value innovation”

Example Project Overview section:

FILE · CLAUDE.md 

          ## Project Overview
          A local-first VLM workbench powered by LM Studio running Qwen 3.5 VL
          models for image and video analysis tasks: explanation, object segmentation,
          and multi-frame object tracking. Targets ML researchers and computer vision
          engineers who need offline visual dataset inspection tools.

2. Commands (Essential)

Claude Code will run commands on your behalf. If it uses the wrong package manager, forgets a required flag or starts the dev server on the wrong port, things break silently and waste debugging time (tokens). Without this section, Claude Code will guess and frequently use the wrong package manager, skip important flags or invoke scripts that don't exist. A good rule of thumb is that every command you actually run during development and deployment belongs here.

✅ Include
  • Write exact commands with every flag your team always uses
  • Separate dev, build, test, lint, type-check, and service-start commands
  • Include any programmatic service start scripts with their invocation
  • Note non-default ports or host flags that affect connectivity
❌ Avoid
  • Don't use vague descriptions like start the app — always be exact
  • Don't omit flags your team uses consistently (--watch, --host, --passWithNoTests)
  • Don't list every possible script — only what Claude will actually need to run
  • Don't include steps that require manual interactive input Claude cannot provide

Example Commands section:

FILE · CLAUDE.md 

          ## Commands
          - `pnpm dev`          — start Vite dev server (port 5173)
          - `pnpm build`        — production build to /dist
          - `pnpm test`         — Vitest unit tests
          - `pnpm test:e2e`     — Playwright end-to-end tests
          - `pnpm lint`         — ESLint + Prettier check
          - `pnpm typecheck`    — tsc --noEmit

          ## LM Studio server
          Start programmatically via Python before any API calls:
            python scripts/start_lmstudio.py --model qwen/qwen3.5-vl-4b
          This script launches LM Studio, waits for the server on port 1234,
          and loads the specified model. Do not call the API before this exits.
          Health check: GET http://localhost:1234/v1/models

3. Architecture & File Structure (Essential)

Claude Code reads your files but doesn't understand your intent unless told. Give precise instructions on which directories matter, what architecture pattern you use and where new files of each type must go. Without this, Claude Code will often create files in the wrong places and break the consistency that makes the codebase navigable over time.

✅ Include
  • Map every top-level directory with a one-line description of its purpose
  • Name the primary architectural pattern (feature-sliced, layered, MVC, etc.)
  • State explicitly where new files of each type must be placed
  • Note any co-location conventions, such as tests living next to source files
❌ Avoid
  • Don't list individual files — document directories and conventions, not every file
  • Don't include generated or version-controlled-but-ignored directories
  • Don't explain historical reasons for the structure — just state what it is
  • Don't duplicate structure that /init already detected and documented correctly

Example Architecture & File Structure section:

FILE · CLAUDE.md 

          ## Architecture
          Feature-sliced design: each VLM task owns its UI, logic, and types.

          - /src/features/explain           — image/video explanation task module
          - /src/features/segment           — segmentation task module
          - /src/features/track             — multi-frame tracking task module
          - /src/components/ui              — shared primitive components
          - /src/services/lmstudio          — LM Studio API client and request builders
          - /src/services/server            — Python server lifecycle management (start/stop)
          - /src/stores                     — Zustand global state (model config, session)
          - /src/types                      — shared TypeScript interfaces only
          - /src/hooks                      — shared custom React hooks

          Tests live next to the file they test (Segment.test.tsx beside Segment.tsx).

4. Tech Stack & Environment (Essential)

Tells Claude exactly what tools, versions, and configuration exist so it never reaches for an uninstalled library, uses a deprecated API, or misses a required environment variable. A project using React 19 + Zustand + TanStack Query has very different idioms from one using React 18 + Redux + SWR. Wrong assumptions produce uncompilable or inconsistent code.

✅ Include
  • List every major dependency with its version where version affects idioms
  • Specify the runtime and language versions (Node, Python, etc.)
  • Document all environment variables: name, purpose, default value, and where they live
  • State the CSS approach clearly — utility framework, modules, or CSS-in-JS
❌ Avoid
  • Don't list transitive or dev-only dependencies unless they affect Claude's output
  • Don't include tool versions for linters and formatters unless they change behavior
  • Omit environment variables — Claude will skip validation logic without knowing them
  • Don't duplicate what package.json already makes obvious if Claude can read it

Example Tech Stack & Environment section:

FILE · CLAUDE.md 

          ## Tech stack
          - React 19, TypeScript 5.4, Vite 5
          - Zustand (global state), TanStack Query v5 (server state + caching)
          - Tailwind CSS v4, shadcn/ui components
          - Playwright (E2E), Vitest (unit)
          - Python 3.11               — scripts/start_lmstudio.py manages LM Studio lifecycle
          - LM Studio local API       — OpenAI-compatible at http://localhost:1234/v1

          ## Environment variables (.env.local — gitignored, .env.example is canonical)
          - VITE_LMS_BASE_URL         — LM Studio base URL (default: http://localhost:1234/v1)
          - VITE_LMS_MODEL_ID         — model string (e.g. qwen/qwen3.5-vl-4b)
          - VITE_MAX_IMAGE_MB         — max upload size in MB (default: 10)
          - LMS_STARTUP_TIMEOUT_S     — seconds to wait for server ready (default: 30)

5. Coding Conventions (Essential)

Encodes how you or your team writes code. It includes naming conventions, state management approaches, error handling strategies and component structures. This prevents Claude Code from introducing patterns that are inconsistent with the rest of the codebase. Without this section, Claude Code will mix async styles, invent its own naming conventions, and write components that look foreign next to every file it didn't author. Consistency is structural, not cosmetic.

✅ Include
  • Specify naming conventions for files, components, hooks, and services
  • State your preferred async pattern and make it exclusive (async/await only)
  • Define where and how errors are caught, surfaced, and reported
  • Note the import ordering convention if your linter enforces a specific style
❌ Avoid
  • Don't list rules already enforced by ESLint or TypeScript — Claude will see the errors
  • Don't specify formatting preferences handled entirely by Prettier
  • Don't state obvious defaults like 'use const where possible'
  • Don't write a full style guide — focus only on patterns where Claude Code must make a choice

Example Coding Conventions section:

FILE · CLAUDE.md 

          ## Coding Conventions
          - Components: PascalCase files and named exports (ImageExplainer.tsx)
          - Hooks: camelCase prefixed with 'use' (useModelStatus.ts)
          - Services: camelCase, one concern per file (lmStudioClient.ts)
          - Async: always async/await — no .then/.catch chains anywhere
          - Errors: catch in the service layer, surface via toast (useToast hook)
          - State: Zustand for global (model config, session); useState for local UI only
          - Props: destructure at the top of every component; avoid prop drilling > 2 levels
          - Exports: named exports everywhere except Next.js / router page components

6. Workflows (High Value)

Prevents Claude from implementing a feature without tests, creating an API call without error handling, or shipping UI without a loading state. Coding agents are stateless, they don't recall that last time they forgot to update types or skipped the empty state. Workflows are the checklist that makes repeatable tasks actually repeatable.

✅ Include
  • Write numbered steps for any task that touches more than two files
  • Include explicit confirmation checkpoints before risky or irreversible actions
  • Add a workflow any time Claude has previously missed a required step
  • Specify ordering where sequence matters. Types before implementation, tests before merge
❌ Avoid
  • Don't create a workflow for every possible task. Only recurring, multi-step ones
  • Don't make steps so granular they read as noise ('open the file', 'save your changes')
  • Don't duplicate information that already lives in the Commands section
  • Don't write workflows for simple, single-file edits

Example Workflows section:

FILE · CLAUDE.md 

          ## Workflows

          ### Adding a new VLM task feature
          1. Create /src/features// with index.tsx, types.ts, hooks.ts
          2. Add LM Studio prompt builder in /src/services/lmstudio/.ts
          3. Register the task route in /src/router.tsx
          4. Add task entry to the sidebar nav in /src/components/Sidebar.tsx
          5. Write unit tests for the prompt builder
          6. Write a Playwright E2E test for the happy path
          7. Confirm with me before committing

          ### Modifying the LM Studio client or Python start script
          1. Check existing types in /src/types/lmstudio.ts before changing anything
          2. Update types first, then implementation
          3. Update the MSW mock in /src/services/lmstudio/__mocks__/ to match
          4. Re-run pnpm test to verify nothing regressed

7. UI and Design Rules (High Value)

Tells Claude Code how the interface must look and behave, which component library to reach for, what spacing scale to use, how dark mode is implemented, and what accessibility baseline is required. Without UI rules, Claude Code invents its own spacing, uses raw HTML elements instead of your design system, skips ARIA attributes, and produces screens that look inconsistent with every other page.

✅ Include
  • Name the component library and define when to use it versus raw HTML
  • Specify the spacing and sizing scale used throughout the UI
  • Define how dark mode is implemented at the code level
  • Set the minimum accessibility baseline and list what is always required
❌ Avoid
  • Don't list individual Tailwind classes. Define conventions and non-obvious rules
  • Don't describe the desired visual aesthetic in subjective terms ('feel modern')
  • Don't include brand guidelines unless Claude is expected to pick colors or write copy
  • Don't repeat constraints the component library already enforces automatically

Example UI and Design Rules section:

FILE · CLAUDE.md 

          ## UI and design rules
          - Use shadcn/ui components first; only build custom if shadcn lacks it
          - Spacing: Tailwind scale only — no arbitrary values (never p-[13px])
          - Dark mode: Tailwind dark: prefix + CSS variables from the shadcn theme
          - Every interactive element needs a visible focus ring (ring-2 ring-offset-2)
          - Loading states: Skeleton components only — no raw spinners or blank panels
          - Empty states: every list or grid needs an EmptyState component with an action
          - Task result panels: fixed split layout — input left, output right, 50/50
          - Errors: always show inline message + retry button, never just a toast alone

8. Domain Terminology (High Value)

Defines vocabulary specific to this project or domain. Prevents Claude Code from using the wrong terms, editing the wrong entities or introducing naming inconsistencies across the codebase. Terms like 'session', 'prompt', 'mask' and 'confidence' mean different things in different systems. Without definitions, Claude defaults to generic interpretations that may not match your data model.

✅ Include
  • Define any term that carries a project-specific meaning distinct from its common use
  • List entity aliases where the code name differs from the UI or domain name
  • Clarify abbreviations used in variable names and file names
  • Include the precise casing or spelling of proper nouns (product and tool names)
❌ Avoid
  • Don't define standard industry terms that Claude Code already understands well
  • Don't list every variable name — only those that are ambiguous or context-dependent
  • Don't include jargon that is universally understood in the relevant technical field
  • Don't over-explain — one concise line per term is sufficient

Example Domain Terminology section:

FILE · CLAUDE.md 

          ## Terminology
          - Session — one user upload (image or video) plus all its associated task results
          - Task — a single VLM operation: explain, segment, or track
          - Result — structured output from a completed task (text, masks, or bboxes)
          - Mask — binary pixel map output from the segmentation task
          - BBox — bounding box (x, y, w, h) used in tracking task output
          - Prompt — the text instruction sent to the LM Studio model (not a UI input)
          - ModelId — the LM Studio model identifier string
          - Confidence — the model's self-reported certainty score (0–1 float)
          - Frame — a single decoded video frame sent for per-frame VLM inference

9. Testing and Quality Bar (High Value)

Defines what "done" means for quality. Sets the testing strategy, which layers get which test types and what every new feature must cover before it is considered complete. Claude Code will skip tests unless they are explicit requirements. Without a stated quality bar, test coverage is inconsistent. Some features get tested, while others don't.

✅ Include
  • State which code layers require unit tests versus integration or E2E tests
  • Specify the mocking approach for external services and APIs
  • List the minimum scenarios every new feature must cover (happy path, error, empty)
  • Name the test runners and any coverage commands that exist
❌ Avoid
  • Don't demand a specific coverage percentage unless your CI actually enforces it
  • Don't list test utility APIs — Claude can read existing test files to learn conventions
  • Don't require tests for generated boilerplate or thin wrappers around external libraries
  • Don't over-specify test structure — let Claude follow the conventions it sees in existing tests

Example Testing and Quality Bar section:

FILE · CLAUDE.md 

          ## Testing
          - Unit tests: all service-layer functions and custom hooks (Vitest)
          - E2E tests: every new task feature needs a Playwright happy-path test
          - No unit tests required for: shadcn/ui wrappers, purely presentational components
          - Mock LM Studio API using MSW handlers in /src/mocks/handlers.ts
          - Mock the Python server start script with a stub that resolves immediately
          - Every feature must cover: happy path, loading state, API error, empty result
          - Run pnpm test before committing — CI blocks on failures

10. File and Component Placement Rules (High Value)

A more specific extension of Architecture. Gives Claude Code the explicit decision rules for where to create new files, preventing gradual structural drift as the codebase grows. 'Put it somewhere logical' leads to features in /components, utilities in /features, and types scattered across every folder. Explicit placement rules make structure self-enforcing over time.

✅ Include
  • Define placement rules as decision conditions — 'if X then Y'
  • Specify when something belongs in a shared location versus a feature folder
  • State the rule for when a new feature directory should be created
  • Document any path aliases and what they resolve to
❌ Avoid
  • Don't repeat the directory map from the Architecture section — add the decision logic
  • Don't write rules for files Claude would never create (config files, lock files)
  • Don't make every edge case its own rule — a few clear principles cover most cases
  • Omit path aliases — Claude will write incorrect import paths without them

Example File and Component Placement Rules section:

FILE · CLAUDE.md 

          ## File placement
          - New feature: create /src/features// when it will have 3 or more related files
          - Component used by one feature only: co-locate inside that feature folder
          - Component used by two or more features: /src/components/ui/
          - Hook used by two or more features: /src/hooks/
          - LM Studio request builder: /src/services/lmstudio/.ts
          - Types used by one feature: /src/features//types.ts
          - Types used by two or more features: /src/types/.ts
          - Path alias @/ maps to /src/ — always use it, never use relative ../
          - Test file: same directory as the source file, same name + .test.tsx

11. Custom Tools and Scripts (Situational)

Documents team-built scripts and generators that Claude will not discover on its own. Without this, Claude Code writes things from scratch that the team has already built and maintains. Scaffolding scripts, seed utilities, and dev automation tools exist because someone solved a recurring problem. Claude Code doesn't know they exist unless explicitly told.

✅ Include
  • List each script with a one-line description of its purpose
  • Show the most common invocation with a concrete example
  • Mention the --help flag if the tool is self-documenting
  • Include code generators Claude should use instead of writing boilerplate by hand
❌ Avoid
  • Don't document widely known third-party CLIs (git, docker, curl)
  • Don't reproduce full documentation inline — just enough to know when and how to invoke
  • Don't list setup scripts that are only ever run once during initial project onboarding

Example Custom Tools and Scripts section:

FILE · CLAUDE.md 

          ## Scripts
          - pnpm run scaffold:feature  — creates /src/features// with boilerplate files
          - pnpm run mock:server — starts MSW mock server for fully offline development
          - pnpm run seed:sessions — populates IndexedDB with sample session data for testing
          - python scripts/start_lmstudio.py --model — starts LM Studio, loads model,
            waits for server readiness; use --timeout 60 for large models
          - python scripts/stop_lmstudio.py — gracefully shuts down the LM Studio process
          - scripts/export-masks.ts — exports segmentation masks to PNG
            Usage: npx tsx scripts/export-masks.ts --session --output ./out/

12. Gotchas and Warnings (Situational)

The things you would tell a new developer on their first day. Covers edge cases, fragile integrations, and constraints that are not visible from reading the code, but cause real pain when missed. External services and local tooling often have undocumented quirks. Without recorded gotchas, Claude Code, and every new developer for that matter, hits the same walls repeatedly.

✅ Include
  • Document fragile areas where changes have non-obvious downstream effects
  • Warn about quirks specific to external services or local tooling your project uses
  • Flag files or patterns that Claude must never modify autonomously
  • Note environment-specific behaviors that differ between dev and production
❌ Avoid
  • Don't list every known bug — only structural warnings relevant to Claude's work
  • Don't include issues already caught by linters, type errors, or automated tests
  • Don't add so many warnings that every section becomes noise
  • Don't document temporary workarounds without noting that they are temporary

Example Gotchas and Warnings section:

FILE · CLAUDE.md 

          ## Gotchas
          - LM Studio reports HTTP 200 from /v1/models even when no model is loaded;
            always use scripts/start_lmstudio.py which waits for model readiness
          - Images must be base64-encoded in the message content array;
            URL references do not work with the local LM Studio server
          - Qwen 3.5 VL returns segmentation masks as RLE-encoded JSON strings,
            not raw pixel arrays — decode using /src/utils/rleDecoder.ts
          - Video frames must be extracted client-side via ffmpeg.wasm before sending;
            LM Studio does not accept raw video file inputs
          - Do NOT modify /src/services/lmstudio/client.ts retry/timeout logic;
            it is tuned for LM Studio cold-start latency on the first inference call
          - The Python start script requires lmstudio-sdk>=0.1.4 — check requirements.txt

13. Safe-change Rules (Situational)

Defines the boundary between what Claude can change freely and what requires human approval first. Essential for agentic sessions where Claude Code may make many sequential file changes. Claude Code in agentic mode can cascade changes across many files. Without boundaries, it may refactor core services, alter API contracts, or touch authentication logic without realising the risk.

✅ Include
  • List the specific files or directories Claude must never modify without asking
  • Define which change types always require human confirmation before proceeding
  • State what counts as a breaking change in the context of your project
  • Apply a blanket 'always ask before deleting any file' rule as a safe default
❌ Avoid
  • Don't make the restricted list too long, it prevents Claude from doing useful work
  • Don't conflate 'risky to change' with 'hard to change' — impact is what matters
  • Don't apply identical restrictions to test files and production source files
  • Don't leave Claude to infer safe boundaries — be explicit

Example Safe-change Rules section:

FILE · CLAUDE.md 

          ## Safe-change rules
          - NEVER modify without asking: /src/services/lmstudio/client.ts,
            /src/stores/sessionStore.ts, any file in /src/types/, scripts/start_lmstudio.py
          - ASK before: deleting any file, changing a component's public props interface,
            modifying an existing Zustand store's state shape, changing LM Studio request format
          - FREE to change: feature-local components, feature-local hooks, test files,
            Tailwind classes, UI copy and labels within existing components
          - Any change to the LM Studio request or response shape is a breaking change — confirm first
          - When unsure whether a change is safe, ask rather than proceed

14. Content and Copy Guidance (Situational)

Defines how UI text is written. This includes voice, capitalization, error message format, and which proper nouns have a canonical spelling. Prevents Claude Code from mixing tone or using inconsistent terminology across screens. Inconsistent labels erode trust in a professional tool. If one screen says 'Analyze' and another says 'Run inference', users question whether they perform the same action.

✅ Include
  • Define the product's voice with a clear descriptor (precise, friendly, technical)
  • Specify capitalization rules for labels, buttons, headings, and error messages
  • Provide a before/after example of how error messages should be written
  • List proper nouns that have a fixed spelling or casing Claude must follow
❌ Avoid
  • Don't write a full content style guide — only conventions Claude will apply in code
  • Don't include marketing or promotional copy conventions
  • Don't specify typographic choices such as font size or line height — those belong in UI rules
  • Don't leave voice undefined and expect Claude to infer it from existing copy

Example Content and Copy Guidance section:

FILE · CLAUDE.md 

          ## Content and copy
          - Voice: technical and precise — this is a professional tool, not a consumer app
          - Sentence case for all labels, buttons, tooltips, and error messages
          - Proper nouns: 'LM Studio' (two words), 'Qwen 3.5 VL' (spaced, not 'Qwen3.5VL')
          - Task names: capitalised as nouns — Image Explanation, Segmentation, Tracking
          - Error messages: state what failed + what the user should do next
            Good: 'Model not loaded. Start LM Studio and load a model to continue.'
            Bad: 'Error: request failed'
          - Empty states: explain why the area is empty + offer one clear next action
          - Loading messages: be specific about the operation, not generic
            Good: 'Analyzing image...' Bad: 'Loading...'

15. MCPs and External Tools (Optional)

If MCP servers are configured, tells Claude Code what is available and when to reach for them instead of writing code or searching manually for information. Without this, Claude won't know to use a configured GitHub or Filesystem MCP even when that would be the most direct path — it will write code to do something a tool can do in one call.

✅ Include
  • List each configured MCP server and what service it connects to
  • Give one concrete example of when Claude should use each server
  • Note if any MCP tool overlaps with a custom script to avoid duplication
❌ Avoid
  • Don't reproduce the .mcp.json configuration here — it already lives in that file
  • Don't document MCP servers that are planned, but not yet connected or authenticated
  • Don't include this section at all if no MCP servers are configured

Example MCPs and External Tools section:

FILE · CLAUDE.md 

          ## MCP tools
          - Filesystem MCP — read sample datasets from ~/datasets/ during development
            Use when the user asks to test with a specific local image or video file
          - GitHub MCP — create PRs, check CI status, fetch issue context
            Use for any git operation that involves the remote repository
          - Playwright MCP — run browser automation for E2E test debugging
            Use when a Playwright test is failing and needs live visual inspection

16. Self-improvement and Memory Rules (Optional)

Tells Claude when and how to propose updates to the CLAUDE.md file itself. Turns it from a document written once into a living record that improves as the project evolves. A CLAUDE.md written at project start goes stale quickly. The best additions come from real mistakes — a gotcha Claude hit, a convention it guessed wrong, a workflow missing a step.

✅ Include
  • Tell Claude to propose additions when it discovers undocumented gotchas or gaps
  • Define the format for proposals — always a draft in a code block, never a silent edit
  • Ask Claude to flag when it is working outside any documented pattern
  • Establish a clear human-approval step before any change is applied
❌ Avoid
  • Don't allow Claude to edit CLAUDE.md autonomously — it must always propose, never act
  • Don't ask Claude to log session-specific preferences that won't generalise
  • Don't make the rules so complex they discourage proposing improvements at all
  • Don't skip the review step — Claude proposals can contain errors too

Example Self-improvement and Memory Rules section:

FILE · CLAUDE.md 

          ## Self-improvement
          - If you discover an undocumented service quirk, propose a Gotchas entry
          - If you place a file that doesn't fit current placement rules,
            flag it and propose a rule update before proceeding
          - If a workflow step was wrong or incomplete, draft a corrected version
          - Always propose CLAUDE.md changes as a markdown code block prefixed with:
            'CLAUDE.md suggestion:' so they are easy to find in the conversation
          - Never edit CLAUDE.md directly — all changes require human review first
