Local CLAUDE.md Blueprint

Local quirks, focus, and temporary hacks

This personal override layer lets you stash machine-specific quirks, current focus notes, temporary hacks, and credential references, without ever polluting your shared CLAUDE.md or Git history.

Discover exactly what belongs in it, see battle-tested examples for both team and solo workflows, and learn power-user tricks that keep Claude Code perfectly aligned with your environment and priorities.

Subsection 5/8 ~5 min read Claude Code ≥ 2.1

The local CLAUDE.local.md file can be best described as your personal, never-commited override layer. It is primarily intended for local env setup, focus notes and temporary hacks that have no place in the shared repository file.

ℹ NOTE
The CLAUDE.local.md is intended to NEVER BE COMMITTED to a repository. It should always be added to the .gitignore. If this is done, it will never appear in your version history, pull requests, or your teammates' checkouts.

Due to never being commited, the local CLAUDE.local.md file is the right home for everything that is personal, secret, machine-specific or transient. This makes the file ideal for working on a joint repository with a team, but also finds its use for developing any code alone that is intended to be pushed to a (semi) public repository.

Working on a shared repository

Most teams maintain a repo-level CLAUDE.md file that describes the project's architecture, conventions and workflow. That file is for everyone. The CLAUDE.local.md file is specifically for you intended. It holds your current tasks, your local machine quirks and whatever temporary context helps Claude Code to support you.

What to include in the CLAUDE.local.md when working on a joint repository with other team members:

  • Local Environment: paths that differ from CI, local service ports, Docker overrides, your specific Python/Node version.
  • Current focus: what ticket or feature you are actively working on. Claude Code will weight suggestions accordingly.
  • Known local issues: broken tests in your branch, a flaky service on your machine, a migration you haven't run yet.
  • Temporary overrides: instructions that contradict the shared file, but only apply during your current sprint or investigation.
  • Run commands: shortcuts, aliases, or scripts specific to your setup that the team file can't assume everyone has.
FILE · CLAUDE.local.md 

          ## Local Environment

          # Service ports differ from docker-compose defaults
          API_BASE=http://localhost:8080
          DB_PORT=5433  # 5432 is occupied by a local Postgres instance
          # Python env
          PYTHON_BIN=/opt/homebrew/bin/python3.12
          # GPU available — prefer GPU-accelerated test paths
          USE_GPU=true
          CUDA_VISIBLE_DEVICES=0,1

          ## Current Focus

          Working on TICKET-4821: refactoring the segmentation pipeline
          to support multi-frame tracking. All new code goes under
          src/vision/tracking/. The public API surface in
          src/vision/segmentation.py must not change.

          ## Known Local Issues

          ! tests/test_video_loader.py — skipped; requires a sample
            video asset not yet downloaded. Run make fetch-fixtures first.
          ! The LM Studio server is running on port 1234 (not 11434).
            Override any hardcoded Ollama defaults when suggesting commands.
          ! Migration 0042_add_tracking_index has not been applied
            locally yet — avoid suggesting queries on that index.

          ## Temporary Overrides

          # Remove when TICKET-4821 is merged
          # Prefer streaming responses during this spike; ignore the
          # repo-level instruction to use batch inference for VLM calls.
          Streaming is preferred for all VLM inference calls right now.
          # Do not suggest adding unit tests for the tracking module yet.
          # Architecture is still in flux; tests are scheduled post-review.
          Skip unit test suggestions for src/vision/tracking/.

          ## Run Commands

          # Start the dev stack (local port overrides baked in)
          make dev-local          # = docker-compose -f compose.local.yml up
          # Run only fast tests (skip GPU and video loader tests)
          pytest -m "not slow"
          # Lint + type check
          ruff check . && mypy src/
          # Apply pending migrations
          alembic upgrade head

Notice how this file answers questions Claude Code would otherwise have to guess: at which port is the API on, which tests to expect to fail, and which shared instruction is deliberately suspended for this sprint. Also, none of this belongs in the commited file as it may disrupt the workflows of other contributors.

Working alone on a repository

Even if you own the entire repo, the local file still serves a distinct purpose. It is the right place for anything you specifically do not want committed, even by mistake. Two categories are most dominant here: Secrets & Credentials and Temporary State.

First, you can use this file to tell Claude Code where secrets live (.env, ~/.config/…), how to reference them in commands and which patterns it should treat as sensitive, without putting such values in a committed file. Note, never put your actual keys in this file, only their references.

Second, the in-progress content. This may include notes about a half-finished migration, a benchmark you're running or a feature behind a flag. Once it's shipped or abandoned, you can just delete the file without having to clean up any git history.

FILE · CLAUDE.local.md 

          ## Credentials & Secrets

          # All secrets are in .env (also git-ignored).
          # When constructing CLI commands that need auth, read
          # from .env or suggest: source .env &&
          Treat any string matching sk-, ghp_, or xoxb- as a secret.
          Never include secret values in code suggestions or commit messages.
          # LM Studio key (local only, no auth required)
          LM Studio API base: http://localhost:1234/v1 — no key needed.
          Qwen2-VL model string: qwen2-vl-7b-instruct


          ## Temporary State
          # Delete when the RL training run is complete
          Currently benchmarking PPO vs SAC on the intercept-v2 env.
          Results land in runs/. Do not suggest changes to reward shaping
          or hyperparameters — treat them as frozen for this experiment.
          The 3D visualizer in sim/viz.py is intentionally decoupled
          from the training loop. Don't suggest merging them.
⚡ Power User Hacks
  • Use it as a sprint scratchpad: start every ticket by writing two or three lines into your local file. Include the goal, the files in scope, what's off-limits. Delete them when the PR merges.
  • Annotate broken test state: note which tests are currently failing and why. Claude stops suggesting fixes for known issues and focuses on the actual task instead.
  • Override verbosity per session: add a line like "Keep explanations brief, I know this codebase" or "Explain every step, I'm learning this stack". Swap it out as needed without touching shared config.
  • Mark divergence from shared instructions: when overriding a rule from the repo-level file, write a comment saying so. Future-you will know it's intentional, not an accident.
  • Keep credential patterns, not values: instead of pasting a token, tell Claude the pattern (Bearer $MYAPP_TOKEN) and where the env file lives. Safe and equally useful.
  • Date your temporary overrides: a comment like (# added 2026-03-27, remove after deploy) makes it easy to audit stale instructions during a cleanup pass.
⚠ WARNING
Don't let it rot. The local file is powerful because it is current. An outdated local file with stale task context or wrong port numbers is worse than no file at all. Remember that Claude Code will most probably act on it. It is best to do a quick review at the start of each new workstream.
ℹ NOTE
When multiple CLAUDE.md files are present, Claude Code merges their contents by loading them alongside each other. There is no strict precedence rule between levels; conflicting instructions depend on how Claude interprets the combined text. This layered system lets you maintain a clean separation between what is universal to you as a developer, what belongs to the project and what is strictly local to your current environment.
