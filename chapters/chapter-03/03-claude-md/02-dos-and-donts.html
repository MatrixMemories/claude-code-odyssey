CLAUDE.md Do’s & Don’ts

How to Make Every Instruction Actually Stick

Mastering your CLAUDE.md is the single highest-leverage move you can make in Claude Code. It shapes every session, every task, and every line of output the model produces.

In this guide you’ll discover the proven do’s and don’ts that separate crisp, reliable agent behavior from bloated, ignored instruction files: why length kills performance, how to keep every rule universally useful, and the ruthless editing habits that keep your CLAUDE.md lean and obeyed.

Subsection 2/8 ~9 min read Claude Code ≥ 2.1

We have established what the CLAUDE.md file is, its purpose within the Coding Agent and high level dimensions that guide the development of the file. Next, we look into the "do's" and "don'ts" of writing an effective CLAUDE.md file. These will help guide the style of the file and provide additional valuable insights into the nature of the file.

The curated list of "do's" and "don'ts" includes:

  • ✅ DO: Debug the CLAUDE.md Like You'd Debug Code

❌ DON'T: Make It a Wall of Text

Keep your CLAUDE.md file as short as possible. A reasonable upper bound is around 300 lines. However, the best CLAUDE.md files are far shorter than that. It is recommended to adopt an ultrashort philosophy: if a line doesn't earn its place, cut it.

This isn't just aesthetic preference, it's grounded in hard research. Frontier LLMs, including Anthropics Claude, can follow at most around 200-300 instructions with reasonable consistency before performance collapses. According to the IFScale benchmark, most models remain stable only up to about 200 instructions, beyond that accuracy drops sharply. The takeaway is blunt: as instruction count climbs, the model begins failing to follow all of them uniformly, not selectively.

What makes this especially important for Claude Code is the layered instruction environment you're already operating in. The Claude models running in the background have a substantial system prompt of their own. On top of that, any plugins, skills, MCP tools, or user turn messages all contribute additional instructions. Your CLAUDE.md sits inside this pile. Every line you add competes with every other instruction in the context window and slowly chips away on the instruction-following quality of the Claude models across the board.

ℹ NOTE
Note that all models suffer from this issue, regardless of attributes like size, reasoning effort or context window. Furthermore, note that the most recent IFScale benchmark results are from July 2025 (around Sonnet 4, Gemini 2.5 and GPT-4.5 time). Since then, models have improved and can handle more than 200 instructions, but community estimates still put the point of collaps well below 500.

✅ DO: Keep Instructions Universally Applicable

LLMs perform best when the context around a task is focused and directly relevant to that task. The CLAUDE.md file is loaded into every Claude Code session regardless on what you are working on. This means that all irrelevant instructions don't just sit quietly, they actively dilute the signal.

Aim to make every instruction in your CLAUDE.md applicable to as wide a range of tasks as possible within your project. Avoid task-specifics that only matter in narrow circumstances. For example, a detailed database schema is just dead weight when Claude Code is helping you write frontend components. It eats context, adds to the instruction count and does nothing for the current task.

The Anthropic team has made this dynamic explicit: if a Claude model detects that a CLAUDE.md file is poorly suited to the current task, it may choose to downweight or flat out ignore it. This behavior was built in deliberately, because ignoring a bad CLAUDE.md yields better results than following it. The conclusion: the more task-irrelevant content you include, the higher the chance your CLAUDE.md context will clash with the current task and thus be entirely ignored by the LLM.

ℹ NOTE
The practical implication: write instructions in your CLAUDE.md file that would improve the output of the Claude model on any session, not just the one you're currently working on. If an instruction only matters for one narrow context, it belongs in a comment, a downstream skill or something third, not in the root CLAUDE.md file.

❌ DON'T: Patch Problems with Reactive Hotfixes

One of the most common antipatterns in Claude Code workflows is reactive hot-fixing: Claude Code does something you didn't want, so you bolt a "NEVER do X" instruction onto the end of your CLAUDE.md and move on. If you repeat this a few times, you will have a file bloated with hotfixes that don't address the underlying cause.

Every hotfix you add is another instruction added to an already crowded context. Worse, these rules tend to be written in the heat of frustration, imprecise, overly broad or contradictory with earlier instructions. They often accumulate faster than they are cleaned up, turning your CLAUDE.md file into a bloated list of band-aids. Over time, this weakens the LLM’s ability to follow the instructions consistently, eventually reaching a point where the file is effectively ignored.

When Claude Code does something unexpected, the right first move is to ask why. Was the task ambiguous? Was a type missing? Was the project structure sending the wrong signal? More often than not, the real fix is upstream: a clearer workflow step, a better-defined interface, a more sensible file organization.

ℹ NOTE
Reserve CLAUDE.md rules for durable, project-wide truths. If a rule exists because of one bad session and addresses a symptom rather than a cause, it probably doesn't belong there. Your CLAUDE.md should feel like a thoughtful constitution, not a criminal code full of after-the-fact amendments.

✅ DO: Structure for Clarity and Front-Load What Matters

A CLAUDE.md file that contains good rules, but presents them poorly is still a bad CLAUDE.md file. Structure is not optional or good to have, it directly affects which instructions Claude Code follows and how reliably. A good rule of thumb is to create a CLAUDE.md file that follows a clear, scannable structure so that Claude Code can instantly grasp what matters most.

Start with the most important instructions. When it comes to instruction-following, LLMs have a strong primacy/recency bias: they attend more strongly to content at the beginning and end of the full context window. The Claude system prompt occupies the very top of the context window, what follows is the CLAUDE.md file. That means that the CLAUDE.md file is prime real estate for your instructions, especially the beginning of the file. Don't bury critical behavior behind a long project description or a preamble that could have been cut.

Beyond the placement and structure, it is important to be extremely clear and concise on every rule. Furthermore, every rule should be verifiable, so that it can be followed correctly or violated detectably. Avoid using vague language like "write clean code", "follow best practices" or "be thoughtful about error handling" — these are aspirations, not instructions. Prefer formulations that can, in principle, be checked: "do not use any in TypeScript" or "run tests before marking a task complete".

Do not reference external files with phrases like "see file X for conventions". Claude Code will almost never fetch and read that file. If something is important enough to be included, include it directly. If it's too long to include directly, it's probably too specific to belong in the CLAUDE.md at all. If you must reference an external file use @ instead.

ℹ NOTE
Keep the document non-verbose. Say a rule once, say it plainly and stop. Repetition and hedging language ("please try to", "generally speaking, you should") are not free. They consume tokens and dilute the instruction's weight without providing any upside.

❌ DON'T: Use It as a Linter or Style Guide

For those that don't know, a linter is a static analysis tool that checks your code for formatting errors, style violations and common mistakes. Linters are automated and cheap tools that don't consume any context in your Claude Code sessions. Forcing your Claude Code Agent to replicate that job through the CLAUDE.md is a mistake on every level.

Code style guidelines — indentation rules, naming conventions, import ordering, bracket placement — inevitably balloon into dozens of instructions and example snippets that dilute the model's attention. LLMs are masterful in-context learners: if your existing codebase already follows clear patterns, Claude Code will tend to match those patterns simply by observing the existing code. You don't need to describe what it can infer.

The correct approach is to keep linting and formatting outside the model entirely. Use a hook that runs your formatter and linter after each Claude Code action and surfaces errors as structured output for Claude to fix, rather than asking Claude to detect formatting problems from scratch. Tools like Biome (for web projects) and Ruff (for Python) are particularly well-suited here, because they can auto-fix a large class of issues before Claude Code ever sees them. For the issues these tools can't auto-fix, presenting the linter's error output to Claude is far more reliable than asking Claude to remember a style rule it read once in a config file.

ℹ NOTE
The division of labor is simple: let deterministic tools handle deterministic rules, and let Claude handle the parts that actually require reasoning.

✅ DO: Start Minimal and Iterate

The CLAUDE.md file is not a one-and-done document you set and forget. On day one, your CLAUDE.md file should be short, even uncomfortably short if you're used to writing comprehensive documentation. That is your starting point. You do not know yet which instructions will matter, which the model will follow naturally without prompting, and which will prove too narrow to keep.

The best practice is to let your file evolve organically as your project grows or as you work on additional parts of an already existing project. Only respond to real observed behavior. Ideally, you should update the CLAUDE.md file after every few sessions, rather than every few minutes. When something in a session works well — a phrasing that led to the right behavior, a constraint that prevented a recurring mistake — capture it. When something stops being relevant, delete it.

Deletion is as important as addition. Apply a ruthless deletion discipline: if a section is no longer active, if a rule was added for a problem that no longer exists, or if an instruction covers something Claude already does correctly by default, cut it. A line that doesn't change behavior is just noise that makes every other line slightly less likely to be followed.

ℹ NOTE
You can enlist Claude Code itself in this process. After every couple of sessions or at any within a conversation, you can ask Claude Code to review the current CLAUDE.md and propose updates, additions, or removals. Include your observations in the prompt when asking Claude Code for the review as well, especially if there are notes from previous sessions. Treat the file as living code, with its own feedback loop.

❌ DON'T: Auto-Generate with /init and Call It Done

Running /init is tempting, since it produces a CLAUDE.md file instantly from your project structure, so you can get to work immediately, and it feels like you've done the responsible thing. Don't fall into this trap. The autogenerated CLAUDE.md file is only a rough starting draft, and it is often a poor one, so never treat it as final.

The CLAUDE.md file is the highest-leverage configuration point in the entire Claude Code harness. Every line of the file shapes the context of every session, every task and every output the LLM produces. The full potential of the leverage is wasted on an auto-generated file. It typically includes noise from directory scans (irrelevant file paths, boilerplate descriptions), misses project specific conventions that aren't inferrable from the file tree, and produce a structure that doesn't reflect how the project is actually worked on.

ℹ NOTE
If you end up using /init at all, treat the output as raw material. Read every line critically, cut what doesn't apply, rewrite what's imprecise, and add the things that actually matter to how you work. A few minutes of deliberate editing will outperform the generated output by a wide margin. Shortcuts here compound into worse sessions over the life of the project.

✅ DO: Debug the CLAUDE.md Like You'd Debug Code

If your CLAUDE.md file isn't producing the behavior you expect, that's a bug and it should be treated like one. Staring at the file and guessing won't fix it, what you need is a feedback loop.

Start by reviewing agent logs from recent sessions. Look for recurring mistakes, failed tool calls, misaligned outputs, or patterns of behavior that your CLAUDE.md should be preventing but isn't. Often the culprit is one of three things: a missing rule, an ambiguous rule that the LLM interpreted differently than you intended, or a contradiction between two rules that leaves the model no good path. You can feed these logs directly to Claude Code and ask it to diagnose which instruction failed and why.

Once you've identified a suspect rule, test it deliberately. Design a small task that creates pressure on that specific instruction, something that would tempt the model to violate it if the rule weren't working. Observe the output. If Claude Code breaks the rule, diagnose immediately: is the instruction too vague? Too buried? In conflict with another instruction? Rewrite it and re-test.

ℹ NOTE
The standard is not "The LLM usually follows this rule." The standard is: the LLM follows this rule even under pressure, even at the end of a long session, even when the user's task seems to pull in another direction. A rule that only holds in easy cases isn't worth the tokens it costs. Writing it is meaningless, unless Claude obeys it when it counts.

❌ DON'T: Include Empty, Aspirational, or Deprecated Sections

Every section of your CLAUDE.md file consumes attention from the LLM and tokens from your context window. Empty sections consume both while providing nothing in return. An empty "Testing Strategy" heading tells Claude Code nothing about how to test your code, it just signals an intention you haven't acted on yet.

Aspirational sections are a close relative: content that describes where the project is going rather than where it is. If a section exists to give Claude Code a sense of future direction, it's almost certainly adding irrelevant context. The right place for project vision is a brief, grounded project description at the top of the file. Anything that doesn't fit there is not relevant to the current task.

Deprecated sections are the most insidious. These are instructions that were genuinely useful during an earlier phase of development. For example, some backend scaffolding rules that guided a month of intensive work but are no longer active concerns. They linger because deletion feels risky: what if you need them again?. In practice, they dilute the file's relevance and add to the instruction load for work they have no bearing on. If a completed module has its own conventions worth preserving, document them in that module's local context (a subdirectory CLAUDE.md or an inline README), not in the root file that governs every session.

ℹ NOTE
The rule is simple: if a section isn't actively shaping Claude's behavior on current work, it shouldn't be there.
