1. @ Imports in CLAUDE.md

External files can be pulled into your CLAUDE.md using the @ syntax, the same way you reference files in a Claude Code prompt. An import can appear anywhere in the file: mid-sentence, in a list item, or on its own line. There are no structural constraints on placement.

Path Resolution

Both absolute and relative paths are valid. Relative paths resolve relative to the file containing the import statement, not to the current working directory.

For example, if your project CLAUDE.md lives at /projects/my-app/CLAUDE.md and contains the line:

CLAUDE.md 

          @docs/architecture.md

…it will look for /projects/my-app/docs/architecture.md, not ./docs/architecture.md relative to wherever you launched Claude Code from. An absolute import like @/home/user/.claude/personal.md resolves to that exact path on disk, regardless of where the project lives.

Recursive Imports and Depth Limit

Imported files can themselves contain @ imports. CLAUDE.md can import @docs/architecture.md, which imports @docs/database.md, which imports deeper context, up to five levels deep. This makes recursive imports well-suited for building layered, modular configuration hierarchies where each file owns a specific slice of context.

ℹ NOTE
The first time Claude Code encounters external imports in a project, it shows a one-time approval dialog listing the specific files that would be loaded. If you approve, they are loaded. If you decline, the dialog does not resurface and those imports remain disabled for that project. This is a per-project decision. Review it carefully, because there is no follow-up prompt.

How Are Imports Loaded?

This is the most important thing to understand about @ imports: they are not lazy references, and they are not loaded on demand. When the session starts, every referenced file's full content is expanded and inserted into context alongside the CLAUDE.md that references it. It behaves exactly as if you had copy-pasted the file contents directly into CLAUDE.md at that position.

@ imports are not a dynamic lookup. Claude does not fetch files mid-conversation when they seem relevant. The entire expanded content is present from the first token of the session.

The claim that @ imports are loaded on demand surfaces regularly on blogs and social media. It is completely false.

However, what tends to happen is that Claude re-reads @-mentioned files during the conversation. This behavior fuels the myth, but the explanation is mundane:

  • The loaded file is a static snapshot taken at session start. If the file changes on disk during the session, Claude needs to re-read it to get the updated version.
  • Claude treats the on-disk file as the source of truth. Its default agentic loop essentially says: "I should verify or pull the latest version before relying on this information."
  • After /compact execution or a lot of messages in long-running session, Claude's effective working memory of the initial import can feel stale. Re-reading is its way of refreshing the exact wording without relying solely on the initial context block.

This re-reading has a cost (additional tokens) but also an upside: the file is reinserted fresh into the active attention window, improving the chance that its instructions are followed.

If you want to suppress this behavior and save tokens, add an explicit "Import & Memory Rules" section to your CLAUDE.md telling Claude not to re-read imported files unless it has reason to believe the content has changed. For example:

CLAUDE.md 

          ## Import & Memory Rules

          - Treat @-imported files as static instruction text, inlined at session
            start. Their content does not change during a session.
          - Never re-fetch or re-read an imported file mid-session. If content
            seems stale, tell the user — do not attempt to reload it yourself.

Two Formatting Notes Worth Knowing

@ imports in code spans and blocks are ignored. Writing @path/to/file inside a backtick span or a fenced code block does not trigger an import. This lets you document or discuss import paths without accidentally pulling files into context.

Heading levels are not adjusted. When you import a file that contains its own headings, those headings are inserted as-is, at their original level. If your imported file opens with # Main Conventions, it becomes a top-level heading in the final document — even if it is logically a subsection of your CLAUDE.md. There is no automatic heading-level adjustment. Design your imported files with this in mind: if they are meant to act as subsections, use ## headings inside them rather than #.

What @ Imports Are Not Designed For

@ imports are not a mechanism for progressive disclosure. All referenced content is injected at session start, every import always costs tokens, always, whether or not the current task touches that context.

That said, you can still structure your CLAUDE.md around imports using conditional language that tells Claude when to pay attention to the expanded content. For example:

CLAUDE.md 

          When working on the database layer, refer to @docs/database-conventions.md
          for schema and migration rules.

This does two things: it helps Claude know when to weight that content, and if the re-reading behavior is active, it will pull the file back into the active attention window when you are working on a matching task.

The Two Original Use Cases

  • 1. Breaking up large CLAUDE.md files for maintainability. 400-line CLAUDE.md is hard to navigate and hard to update. Splitting it into focused files — one for conventions, one for architecture context, one for testing rules — makes each file easier to own and edit independently.
  • 2. Avoiding duplication of existing project docs. Your project already has a README, a package.json, a schema file, an architecture document. Without imports, you either paste summaries of all that into CLAUDE.md (now you have two sources of truth to maintain and keep in sync) or Claude works without that context. Imports let you reference the canonical files directly.

Emerging Third Use Case: Replacing CLAUDE.local.md

A third pattern has become increasingly prominent: using @ imports as an alternative to CLAUDE.local.md for personal, machine-specific configuration.

Claims that CLAUDE.local.md has been retired in favor of imports are appearing more frequently. As of early April 2026, CLAUDE.local.md is still fully functional and has not been deprecated. The motivation for the shift is practical rather than prescriptive.

The problem with CLAUDE.local.md is that if you work across multiple git worktrees, the local file only exists in one of them. Your personal preferences, local tool paths, and machine-specific rules are siloed to a single worktree.

If you use a home-directory import instead, all worktrees share the same personal instructions automatically. A minimal setup looks like this in your project's CLAUDE.md:

CLAUDE.md 

          ## Personal Overrides
          <!-- Load developer-specific local preferences. This file is not committed. -->
          @~/.claude/personal.md

The file at ~/.claude/personal.md lives outside the repository entirely, survives across all worktrees, and stays out of version control. For teams, each developer maintains their own, and the project CLAUDE.md simply declares where to look.

2. CHANGELOG.md and the project-notes Directory

Earlier in this tutorial, we covered removing obsolete and deprecated sections from CLAUDE.md. Completed work, abandoned experiments, and resolved decisions do not belong there. They dilute the signal and clutter the context.

But there are two categories of project knowledge that are genuinely valuable to Claude Code, have no natural home inside CLAUDE.md and are routinely lost between sessions:

  • Project history: what changed, when, and why.
  • Dead ends: failed approaches, discarded architectures, and the explicit reasons they were abandoned.

The solution that has emerged and is explicitly recommended by Anthropic, is a CHANGELOG.md file combined with a project-notes/ directory.

Setting Up CHANGELOG.md

Create a file called CHANGELOG.md in your project root, alongside your CLAUDE.md.

Example structure for the CHANGELOG.md file:

CHANGELOG.md 

          ## Current Status
          [One paragraph: where the project stands right now, what is working,
          what is broken, and what the immediate next step is.]

          ---

          ## [YYYY-MM-DD] — Brief Label
          ### Added / Changed
          - Feature or task description (files affected: src/api.py, tests/test_api.py)

          ### Decisions
          - Chose SQLite over Postgres for local dev — avoids docker dependency,
            acceptable for expected load.

          ### Failures
          - Attempted streaming response via Server-Sent Events — CORS headers
            blocked by nginx in current config. Reverted. Blocked until nginx is
            configurable.

          ### Tests
          - test_auth.py: all passing
          - test_stream.py: 2 failing (see Failures above)

          ### Gotchas
          - Hot-reload breaks if you edit config.py while the dev server is running.
            Restart required.

Each session (or coherent group of targeted changes) gets its own dated section. The header date uses YYYY-MM-DD format. The label after the dash is a short, human-readable description of what the session was about.

Section Contents

Placing a Current Status section at the very top of the file is a quite important part of the CHANGELOG.md file. It gives Claude a front-loaded, overview of where the project stands.

Inside each session entry, include the following subsections as applicable:

Subsection What Goes Here
Added / Changed Completed features or tasks, with the files changed. Use Added for new things, Changed for modifications to existing ones.
Decisions Architecture or implementation choices, with the reason.
Failures Failed approaches with explicit reasons. Highest-value content in the file.
Tests Which tests pass or fail at this checkpoint.
Gotchas Known regressions, workarounds, or hacks currently in the codebase.

The Failures section deserves emphasis. Without an explicit record of what was tried and why it did not work, Claude will attempt the same thing again in the next session. This is one of the most expensive failure modes in long-running projects — not just in tokens, but in time and frustration.

Managing Growth: Periodic Compression

As the project progresses, CHANGELOG.md will grow. A long, dense file costs tokens and buries the recent, relevant content under layers of old history.

The practical solution is periodic compression. When the file becomes unwieldy, archive everything older than the current sprint or milestone into a ## History section at the bottom of the file. Condense old session entries into their most important items — relevant decisions that still hold, failures that are still worth remembering, gotchas that have not been resolved. Items that are no longer relevant (a failure from a discarded approach, a decision that has since been reversed) should simply be dropped rather than carried forward.

ℹ NOTE
You can look up the CHANGELOG.md from the clax project by Anthropic at: GitHub - Clax CHANGELOG.md .

Setting Up "project-notes" Directory

Create a project-notes/ directory in your project root or inside .claude/. This is where deep, per-topic context lives.

Each significant task, feature, or bug gets its own focused Markdown file. These files are where you put:

  • Detailed design context and diagrams
  • Subsystem-specific decisions and rationale
  • Known gotchas and edge cases for that area
  • Checklists of TODO items with checkboxes
  • Links to relevant external docs

In addition to per-feature files, you can create cross-cutting files: e.g. a bugs.md listing open issues, or a tooling-notes.md documenting quirks and decisions around external tools and dependencies.

Connecting It to CLAUDE.md

Reference both CHANGELOG.md and project-notes/ in your CLAUDE.md, without using the @ import syntax. The goal is to give Claude instructions for when and how to interact with these files, not to auto-load them at session start (which would cost tokens every session whether or not you need them).

A minimal CLAUDE.md section might look like:

CLAUDE.md 

          ## Session Protocol
          **Session start:** Read `CHANGELOG.md`. Note Current Status and identify the next task.
          If working on a specific feature or subsystem, read the relevant file in `project-notes/`
          before starting.
          **Session end / significant milestone:** Update `CHANGELOG.md` with a dated entry covering:
          what was completed, decisions made, approaches that failed and why, and next steps. Update
          Current Status at the top. Check off completed tasks in the relevant `project-notes/` file.

The explicit instruction to update "Current Status" is worth including by name, since it is the piece Claude Code must maintain for the whole system to work.

The Session Handoff Workflow in Practice

With CHANGELOG.md and project-notes/ in place, a typical session looks like this:

  • 1. Session starts:

    CLAUDE.md auto-loads. Claude reads CHANGELOG.md per the session protocol, sees Current Status, the last entry, and the immediate next task.

  • 2. Starting a feature:

    You say @project-notes/auth-flow.md — continue from where we left off. Claude reads the feature file, sees checked and unchecked tasks, and picks up exactly where work stopped.

  • 3. During work:

    Claude reads @project-notes/auth-flow.md again if it needs to make a decision related to that area.

  • 4. Completing a task:

    Claude checks off the completed items in the feature file and adds an entry to CHANGELOG.md.

  • 5. Context reset/Next session:

    Nothing is lost. The state lives in the files, not in the context window.

A Key Discipline: Claude Writes, Not Just Reads. If you treat CHANGELOG.md and the feature notes as read-only documentation that you update manually, you will be doing it yourself after every session. The leverage comes from instructing Claude to maintain them as part of its normal workflow. That makes Claude a participant in the project's own documentation, not just a tool that consumes it.

Note that CHANGELOG.md is loaded explicitly (via prompt or instruction) rather than auto-loaded. This is intentional. You @-mention it or any project-notes/*.md file in your prompt when it is relevant to the current task, rather than paying the token cost every session.

Why The CHANGELOG.md Matters?

  • Dead-end prevention. Without a record of failed approaches, Claude will attempt the same thing again in the next session. This is one of the most expensive failure modes in long-running projects.
  • Resumption without re-explanation. At the start of a session, you or Claude can simply say "Read CHANGELOG.md and pick up the next task." Claude immediately has the project's state — what works, what does not, and what comes next — without you reconstructing it from memory.
  • Supervision without presence. In agentic workflows running over hours or days, CHANGELOG.md acts as the log you check when you return to assess what the agent did. It is the equivalent of a commit log that explains why, not just what.

Is This Necessary?

For short, single-session tasks: no.

For anything spanning multiple sessions — a feature, a module, a multi-day project — the answer is effectively yes. Without it, you will either waste time re-establishing context at the start of each session, or Claude will silently drift: re-attempting things that already failed, inconsistently applying earlier decisions, or losing track of what is done.

The pattern is particularly critical in agentic and autonomous work that include long background runs, /loop tasks or multi-agent pipelines. For long-running software projects that span multiple agent sessions, memory files need to be bootstrapped deliberately and maintained systematically, not written ad hoc as work progresses. CHANGELOG.md turns memory into a structured recovery mechanism: each new session can pick up exactly where the last one left off.

3. How the Harness Wraps Your CLAUDE.md

CLAUDE.md may resemble a system prompt, but it is not one. Understanding exactly how it gets consumed by the underlying model explains a lot about why certain CLAUDE.md authoring choices matter more than others.

There are two things to understand: the wrapper that is created around the CLAUDE.md content, and where that wrapped content lives in the API call.

The <system-reminder> Wrapper

When Claude Code delivers your CLAUDE.md to the model, it does not inject the raw Markdown. First, it wraps the content in a <system-reminder> XML tag. The global and local CLAUDE.md files are concatenated with the repository-level file and added inside this tag together.

HARNESS 

          <system-reminder>
          As you answer the user's questions, you can use the following context:

          # claudeMd
          Codebase and user instructions are shown below. Be sure to adhere to these instructions.
          IMPORTANT: These instructions OVERRIDE any default behavior and you MUST follow them exactly as written.

          Contents of /Users/userx/projectx/.claude/CLAUDE.md (project instructions, checked into the codebase):
          [...entire CLAUDE.md file content...]

          IMPORTANT: this context may or may not be relevant to your tasks.
          You should not respond to this context unless it is highly relevant to your task.
          </system-reminder>

The XML wrapper serves two purposes:

  • Structural boundaries: Claude models are fine-tuned on XML-style tags for separating instructions, context, tool outputs, and artifacts. Using a named tag reduces ambiguity and attention drift compared to injecting raw Markdown.
  • Semantic signaling: The tag tells the model: this is high-priority persistent guidance, not a user query or tool output to reply to directly.

The Three Statements Inside the Wrapper

The <system-reminder> tag contains three distinct statements that together define how the model perceives CLAUDE.md content:

  • Statement 1 — The framing prelude: "As you answer the user's questions, you can use the following context:"

    This tells the model that the block below is background material allowed to be referenced, while completing its real task (answering the user, completing the coding task). It positions CLAUDE.md as a resource, not a directive.
  • Statement 2 — The namespace label and override instruction.

    The # claudeMd header acts as a namespace label telling the model where CLAUDE.md content begins. The accompanying instruction declares that these instructions override default Claude behavior and must be followed exactly as written. This gives the model a strong signal that the content inside is authoritative project-specific guidance.
  • Statement 3 — The relevance guardrail: "IMPORTANT: this context may or may not be relevant to your tasks. You should not respond to this context unless it is highly relevant to your task."

    This is the anti-distraction statement. It gives the model explicit permission to deprioritize or ignore CLAUDE.md content if the current task does not make it clearly useful.

The Built-In Contradiction: Statement 2 says "IMPORTANT: These instructions OVERRIDE any default behavior and you MUST follow them exactly as written." Statement 3 immediately says "this context may or may not be relevant to your tasks."

The second statement wins, because it is a meta-instruction about how to treat the first. In practical terms: Claude will ignore the contents of your CLAUDE.md if it decides they are not relevant to its current task.

This has a direct implication for CLAUDE.md authoring. The more content you have in the file that is not universally applicable — old decisions, context about completed features, documentation about areas Claude is not currently working in — the more likely it is that Claude will deprioritize the whole block. The relevance guardrail gives the model the authority to self-decide whether to apply your instructions.

This is a real architectural tension. It has been filed as a bug on the Claude Code GitHub repository multiple times. It is also one of the strongest arguments for keeping CLAUDE.md short, high-signal, and universally applicable to the work Claude is actually doing.

Where the Wrapped Content Lives in the API Call

When Claude Code sends a request to the Anthropic API, the request is structured with three sections: system, tools, and messages.

The <system-reminder> containing your CLAUDE.md is not placed in the system field. It is injected as a fake user message in the messages array. 


      Anthropic API Request
      ├── system[]
      │   ├── "You are Claude Code..."
      │   ├── Task execution rules
      │   ├── Tool usage policy
      │   └── __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__
      │       ├── Git status snapshot
      │       └── (other per-session system content)
      │
      ├── tools[]
      │   └── Read, Edit, Bash, Grep, ...
      │
      └── messages[]
          ├── {role: "user", content: "<system-reminder>
          │       As you answer the user's questions, you can use the following context:
          │
          │       # claudeMd
          │       Codebase and user instructions are shown below...
          │       IMPORTANT: These instructions OVERRIDE any default behavior...
          │
          │       Contents of /path/to/CLAUDE.md:
          │       [YOUR CLAUDE.MD CONTENT HERE]
          │
          │       IMPORTANT: this context may or may not be relevant...
          │       # currentDate
          │       Today's date is 2026-04-09.
          │   <system-reminder>"}     ← Injected as fake user message, isMeta:true
          │
          └── {role: "user", content: "your actual prompt"}

This is a deliberate design choice. The core system prompt stays frozen and static, which is necessary for prompt caching to work and to keep response costs and latency predictable. The per-project CLAUDE.md content is delivered through the messages array, where it still receives high-priority treatment from the model (because of the wrapper and its training) without breaking the cache.

Re-Injection When Files Change Mid-Session

Claude Code's native binary tracks every file it reads or edits in an internal readFileState map. Each entry stores the file content and a timestamp derived from the file's modification time (Math.floor(mtimeMs)).

Before each conversation turn, the binary polls every tracked file. If a file's modification timestamp has changed since the last read, the full file is re-read from disk and injected as a fresh <system-reminder>. This applies to CLAUDE.md itself, if you edit it mid-session, the updated content will be re-injected on the next turn.

Other Content Injected via <system-reminder>

CLAUDE.md is not the only thing delivered this way. The following are also injected using the same <system-reminder> XML tag:

  • Empty To-Do List Reminder
  • File Modified by User or Linter notifications
  • Plan Mode Active status
  • Token Budget / USD Budget tracking
  • Agent Mention context

4. Including Your CLAUDE.md in the System Prompt

For cases where the relevance guardrail is unacceptable — where you genuinely need certain instructions followed every turn without exception — Claude Code offers a set of flags that let you inject content directly into the system prompt. This is the nuclear option for adherence.

The system prompt occupies a fundamentally different position than CLAUDE.md:

  • It is positioned at the forefront of every turn and is never buried in context or compacted away.
  • There is no relevance filter, it is injected as a premise for the entire generation process.
  • During training, models are reinforced to treat the system prompt as highest-priority, non-negotiable rules.
  • Chat templates and special tokens signal to the model that this is authoritative developer-level instruction.

All of this gives system prompt instructions significantly higher adherence and authority than anything delivered via the messages array — including CLAUDE.md.

The Four Flags

All four flags are applied at startup:

Flag Behavior
--append-system-prompt "text" Appends the given text to the end of the existing Claude Code system prompt. All built-in capabilities remain intact.
--append-system-prompt-file ./path/to/file.md Same as above, but reads the content from a file.
--system-prompt "text" Replaces the entire system prompt except tool definitions. Completely removes the built-in instructions. Use with caution!
--system-prompt-file ./path/to/file.md Same as above, but reads from a file.

Example usage:

Bash 

          # Inject CLAUDE.md content into the actual system prompt

          claude --append-system-prompt "$(cat CLAUDE.md)"

          # Or using the file flag directly (cleaner)
          claude --append-system-prompt-file CLAUDE.md

          # Or replace the entire system prompt (dangerous — you lose all built-in behaviors)
          claude --system-prompt-file my-custom-system.md

What You Get by Injecting Into the System Prompt

Instructions placed in the system prompt via these flags are:

  • Always present in the model's attention, every turn, without exception.
  • Cannot be overridden by user messages or CLAUDE.md content.
  • Treated as foundational behavior rather than contextual guidance.
  • Persistent across context compaction.

The Cost: You Break Prompt Caching

Injecting custom instructions into the system prompt has a significant downside: it breaks the prompt cache.

Without going into the full mechanics of caching, the practical consequences are:

  • You will be charged significantly more (or consume your quota faster) because cached tokens are billed at a fraction of the cost of fresh tokens.
  • Response latency will increase.
  • If your injected instructions conflict with or partially duplicate the built-in system prompt, you may introduce subtle behavioral conflicts that produce worse results than the default configuration.

When to Append Instructions to the System Prompt

Use --append-system-prompt (or the file variant) when cost and latency are acceptable and you need:

  • Absolute behavioral constraints that are non-negotiable and should not be subject to any relevance filter — for example, "never commit directly to main" or "always write tests before implementation."
  • Role or persona definitions — if you want Claude to operate as a specialized agent: security reviewer, performance optimizer, strict code reviewer.
  • CI/CD and automation pipelines — in non-interactive -p mode where you cannot supervise, you need rules enforced absolutely.
ℹ NOTE
In all of these cases, append only a small, focused set of hard rules — 3 to 7 lines — that you truly need enforced every turn without exception. The more you inject here, the higher the cost and the greater the risk of conflicts.

When to Replace the System Prompt Entirely

Use --system-prompt (or the file variant) only when:

  • You are building a custom agent from scratch that is not a software engineering assistant and does not benefit from the default coding guidelines.
  • You need complete behavioral control and the default instructions are irrelevant or actively counterproductive.
  • You are doing research or testing on minimal or custom prompts and need a clean slate.
⚠ WARNING
Replacing the system prompt removes approximately 50 carefully designed instructions. Do not do this for a standard software development workflow.

5. Good to Know

CLAUDE.md Is Re-Read After /clear

The /clear command wipes conversation history but does not end the session. Critically, it removes accumulated history while preserving your CLAUDE.md configuration. The CLAUDE.md file — along with all other files loaded at startup, including rules files and global/local files — is reinjected before the first new message in the cleared session.

The practical implication: you can freely use /clear between unrelated tasks to reset the context window entirely without losing your project configuration. Use it liberally.

HTML Comments Are Stripped Before Injection

HTML comment syntax (<!-- -->) is stripped by the Harness from CLAUDE.md before the content is sent to the model. That means HTML comments cost zero tokens and will never interfere with your instructions.

This makes them ideal for maintenance notes — annotations you leave for yourself (or future maintainers) that have no business being in the model's context. Any note in the format <!-- maintenance note: ... --> will be silently removed before injection. Every rule, section, or import in your CLAUDE.md can have a comment that is strictly for human readers.

Re-Reference CLAUDE.md When Adherence Drops

In long-running sessions, adherence to CLAUDE.md instructions can drop noticeably as the conversation progresses and older context is deprioritized. A well-established mitigation is to explicitly @-reference the CLAUDE.md file in your prompt when you notice drift:

Claude Code 

          @CLAUDE.md — check these rules before proceeding.

This forces the file back into the active attention window at that point in the conversation, effectively re-anchoring Claude to the instructions. It works because it re-surfaces the document as a freshly relevant user-turn reference rather than background context that may have been deprioritized by the relevance guardrail.

Shell Commands: Code Fences vs. Prose

Shell commands mentioned in CLAUDE.md should always be wrapped in backticks, not described in plain prose.

CLAUDE.md 

          Run tests with `npm run test`

The backtick signals to Claude that this is a literal executable command, not a description of one. For multi-line commands, use a fenced code block:

CLAUDE.md 

          ```bash
          cp .env.example .env
          npm install
          npm run db:migrate
          npm run db:seed
          ```

Claude treats content inside a code fence as a verbatim command to invoke. A command written in a sentence is a suggestion it might improvise on; a command in a code fence is something it will run exactly as written.

For this reason: make sure these are the correct commands before you put them in CLAUDE.md. Claude will run them verbatim when you ask it to run tests, build the project, or trigger a lint pass. Be specific rather than descriptive.
