Progressive Disclosure

Keeping Context Lean as Projects Grow

As your Claude Code projects grow from simple prototypes into complex full-stack monorepos, a single monolithic CLAUDE.md file quickly turns into a liability—bloating context, burning tokens, and diluting the instructions that actually matter.

Progressive Disclosure solves this elegantly by distributing your rules and conventions across multiple targeted files so Claude Code loads only the relevant guidance for the task at hand.

This section explores the most effective real-world techniques, from hierarchical nested CLAUDE.md files and path-scoped rules to reliable manual methods, giving you practical ways to keep your AI assistant sharp and efficient at any scale.

Subsection 6/8 ~11 min read Claude Code ≥ 2.1

Throughout this section we have hammered on two principles above all others: token discipline and an ultrashort philosophy when writing CLAUDE.md files. The reasoning is simple: Claude Code operates within a fixed context window; can only follow a limited number of instructions reliably, and performs best when that context is packed with usable, directly relevant information rather than noise.

We have also mentioned examples that reinforce these principles, such as removing backend related entries from your CLAUDE.md once you have moved on to frontend work. It sounds clean in theory, but in practice, it is almost impossible to follow.

Most real world projects are not developed in a clean linear sequence. You switch between frontend, backend and other components constantly. You have shared utilities, infrastructure code, tests, a CI configurations, all evolving in parallel. In larger projects it can be genuinely challenging to squeeze everything relevant to even a single component into one CLAUDE.md file. And even in the rare case where you manage it, you have traded one problem for another: context pollution. When Claude Code loads your single monolithic CLAUDE.md, it loads everything, which means that when you are working on a React component, Claude is also sitting on instructions about your pandas data pipeline, instructions about your FastAPI endpoints and your PostgreSQL schema patterns. None of that matters for the task at hand. It just burns context and dilutes the instructions that do matter.

The Solution: Progressive Disclosure

The idea is straightforward: instead of cramming all project instructions into one file, you distribute them across multiple markdown files. The key constraint is that for any given task, Claude Code should read only the instruction files relevant to that task. Irrelevant instructions stay out of context entirely.

There are several ways to achieve this, amongst the most popular are the following five:

  • 1. The Hierarchical Way — nested CLAUDE.md files living inside subdirectories.
  • 2. The Rules Way — scoped instruction files inside .claude/rules/, linked to file paths.
  • 3. The Cowboy Way — a hand-rolled agent_docs directory with semi-manual read enforcement.
  • 4. The Skills Way — misappropriating Claude Code's skills feature (covered in the Skills section).
  • 5. The Hooks Way — on-demand loading driven by programmatic hooks (covered in the Hooks section).

1. The Hierarchical Way: Nested CLAUDE.md Files

If your project has distinct modules, or if you are working in a monorepo, you likely already have a clear directory structure: a root directory with subdirectories like backend/, frontend/, infra/ and so on. The hierarchical approach maps instruction files onto that same structure.

How It Works

You place a CLAUDE.md in the root directory, and additional CLAUDE.md files inside any subdirectories that warrant their own instructions. The behavior is:

  • The root CLAUDE.md is eagerly loaded at the start of every session, unconditionally.
  • Subdirectory CLAUDE.md files are lazy-loaded — they are pulled into context only when Claude Code reads a file inside that directory.

So for a project with backend/ and frontend/ subdirectories, the root CLAUDE.md holds project-wide context (repository layout, shared conventions, tooling), while backend/CLAUDE.md holds backend-specific instructions and frontend/CLAUDE.md holds frontend-specific ones. When Claude is working on an API route, it loads the root and backend files. The frontend instructions never enter the picture.

You can go deeper. If backend/ contains src/, tests/, api/, and data/ subdirectories, each of those can have its own CLAUDE.md. The same lazy-loading rule applies at every level.

Your directory tree might look similar to: 


      project_root/
      ├── .claude/
      │   ├── rules/
      ├── backend/
      │   ├── venv/
      │   ├── src/
      │   │   ├── CLAUDE.md
      │   ├── api/
      │   │   ├── CLAUDE.md
      │   │   ├── __init__.py
      │   │   ├── router.py
      │   │   └── router_health.py
      │   ├── test/
      │   │    ├── CLAUDE.md
      │   ├── CLAUDE.md
      │   ├── pyproject.toml
      │   └── uv.lock
      ├── frontend/
      │   ├── src/
      │   │    ├── CLAUDE.md
      │   ├── CLAUDE.md
      │   ├── package.json
      │   ├── vite.config.ts
      │   └── tsconfig.json
      ├── .gitignore
      ├── .env
      ├── CLAUDE.md
      ├── CLAUDE.local.md
      └── README.md

Path Walking

This is the most important detail to internalize: Claude Code does not just load the CLAUDE.md of the directory it is currently reading, it walks the full path upward to the root and loads every CLAUDE.md it encounters along the way.

If Claude Code reads a file at backend/api/router.py, it will load:

  • backend/api/CLAUDE.md
  • backend/CLAUDE.md
  • CLAUDE.md (root)

All three are returned together as part of the read tool output. This is what makes the hierarchy coherent: lower-level files can assume the higher-level ones have been loaded.

  • The flip side of this is that you must keep the files consistent with each other. Specifically:
    • Do not repeat yourself across levels. If a convention is stated in the root CLAUDE.md, the subdirectory files do not need to restate it.
    • Do not let instructions contradict each other. A rule in backend/api/CLAUDE.md must not conflict with a rule in backend/CLAUDE.md or the root.
    • The total instruction budget still applies across all loaded files. Three short files that are all loaded simultaneously are no different from one medium file.

When Is a Subdirectory CLAUDE.md Actually Loaded?

This is where developers get tripped up, so it is worth stating it explicitly. A subdirectory CLAUDE.md is loaded if and only if Claude Code uses the read tool on a file inside that directory. That is the entire trigger. Nothing else causes it to load:

  • Claude Code creating a new file in a directory (write tool) does not trigger loading.
  • Glob and Grep tools do not trigger loading, even though they are read-adjacent operations.
  • Simply being told to work in a directory does not trigger loading.

The practical implication: instructions about creating brand-new files, building features from scratch, or anything not anchored to an existing file are unreliable if placed only in a subdirectory CLAUDE.md. Claude Code needs to read something in that directory first and in most cases it will. Especially, if you ask it to modify an existing module, it will read the relevant files before writing. But for greenfield work you cannot count on it.

The Burial Problem

There is a subtler issue that goes beyond just whether a file gets loaded: where in the context it lands and when it gets loaded.

Imagine this session:

  • 1. You start a session, provide some planning context, exchange a few messages.
  • 2. You ask Claude Code to update backend/api/router.py. It reads the file, and along with it, backend/api/CLAUDE.md is loaded into context.
  • 3. You continue the conversation. You now ask Claude Code to create a new endpoint file in backend/api/. Claude deems it does not need to re-read any existing files and goes straight to writing.

At step 3, the content of backend/api/CLAUDE.md is buried several messages back in the session context. LLMs have a well-documented primacy/recency bias, that is information at the beginning and end of the context window gets prioritized. Information in the middle gets ignored. Your carefully written instructions are sitting exactly where they are most likely to be disregarded.

If Claude Code re-reads a file from backend/api/, the CLAUDE.md content gets surfaced again into recent context. But if it does not, those instructions may as well not exist for that message.

⚡ Power User Hacks

The mitigation is manual, but reliable: @-mention the CLAUDE.md file in your message. Claude Code will read it as part of processing your message, placing its content in the most recent part of context where it will actually be attended to.

CLI 

            @backend/api/CLAUDE.md Add a new POST /users endpoint following the existing conventions.

Excluding Files You Don't Want Loaded

Sometimes Claude Code will read a file in a directory, but you do not want the corresponding CLAUDE.md to be loaded because it is not relevant to the current task. For example, if you are building a backend API component and Claude Code only needs to know what shape of response the frontend expects — not the full frontend development conventions — you can suppress the frontend CLAUDE.md loading.

This is done via the claudeMdExcludes setting in your project configuration:

json 

          {
            "claudeMdExcludes": [
              "frontend/CLAUDE.md",
              "docs/**"
            ]
          }

Files matching these patterns will be silently skipped even if Claude Code reads files in those directories.

A Note on Stability

As of March 2026, there are known issues with hierarchical CLAUDE.md loading on certain Claude Code versions — particularly the VSCode extension. The CLI is more stable. After setting up nested CLAUDE.md files, verify that they are actually being loaded using the /status command. Do not assume they are working until you have confirmed it.

Why Bother With the Hierarchy?

When it works well, the payoff is significant:

  • Your root CLAUDE.md stays lean and project-relevant, instead of ballooning with module-specific details.
  • Instructions are co-located with the code they govern, making them easier to maintain.
  • A component's instructions stay available and correct even while you are working elsewhere, they do not need to be deleted and rewritten when you context-switch.
  • The amount of irrelevant instruction loaded for any given task is naturally minimized.

2. The Rules Way: Path-Scoped Files

The rules/ directory is Claude Code's built-in mechanism for modular, scoped instruction files. It lives at .claude/rules/ in your project root — exact spelling, case-sensitive.

A parallel personal version can exist at ~/.claude/rules/ for user-level rules you want across all projects.

Structure and File Requirements

  • All files inside rules/ must be .md files. Non-markdown files are silently ignored.
  • Files are discovered recursively, so you can organize them into subdirectories: .claude/rules/frontend/, .claude/rules/backend/, etc.
  • Filenames should be descriptive kebab-case: api-conventions.md, python-style.md, test-patterns.md. Names like rules1.md or r.md are not useful. 

      project_root/
      ├── .claude/
      │   ├── rules/
      │   │   ├── api-conventions.md
      │   │   ├── python-style.md
      │   │   ├── test-patterns.md
      ├── backend/
      │   ├── venv/
      │   ├── src/
      │   ├── api/
      │   ├── test/
      ├── frontend/
      ├── .gitignore
      ├── .env
      ├── CLAUDE.md
      ├── CLAUDE.local.md
      └── README.md

Each rules file has two optional sections:

  • A YAML frontmatter block at the top, delimited by ---.
  • Markdown body content below the frontmatter.

Neither section is strictly required.

Two Modes: Always-Loaded vs. Path-Scoped

The presence of a "paths:" key in the frontmatter is the single deciding factor in how a file is loaded.

Always-loaded (no frontmatter, or frontmatter without paths:)

.md 

          ## Python Style Conventions

          - Always use type hints on function signatures
          - Use pathlib.Path instead of os.path
          - Prefer dataclasses over plain dicts for structured data

Files like this are loaded at the start of every session alongside the root CLAUDE.md. They are useful for keeping the root CLAUDE.md readable and organized, but they do not reduce the total instruction count or solve the context pollution problem. They are essentially an extension of your root CLAUDE.md.

Path-scoped (frontmatter with paths:)

.md 

          ---
          paths:
            - "backend/src/**/*.py"
            - "backend/tests/**/*.py"
          ---

          # Python Style Conventions

          - Always use type hints on function signatures
          - Use pathlib.Path instead of os.path

A file with a paths: key is only loaded when Claude Code reads a file that matches one of the specified paths. This is the mechanism that achieves progressive disclosure.

Note that the paths: parameter uses glob synthax:

Pattern Meaning
** Any number of directories, recursively
* Anything within a single directory segment
? Any single character
{a,b} Matches either a or b
[abc] Matches any character in the set

You can match multiple file types with brace expansion:

.md 

          ---
          paths:
            - "frontend/src/**/*.{ts,tsx}"
            - "frontend/src/**/*.{js,jsx}"
          ---

When Is a Path-Scoped Rule Loaded?

Same constraint as with nested CLAUDE.md files: only when Claude Code uses the Read tool on a file whose path matches the glob pattern. Write, Glob, Grep, and other tools do not trigger loading. If Claude Code never reads a matching file during a session, the rule is never loaded.

Advantages Over Nested CLAUDE.md

Path-scoped rules have one structural advantage over the directory-based approach: they are not constrained by directory boundaries. You can bind a single rules file to paths across multiple unrelated directories:

.md 

          ---
          paths:
            - "backend/**/*.py"
            - "scripts/**/*.py"
            - "tools/**/*.py"
          ---

This instruction file loads whenever Claude reads any Python file across your entire repository, regardless of where it lives. With nested CLAUDE.md files you would need to duplicate content into each relevant subdirectory.

Limitations

The same failure modes that affect nested CLAUDE.md files apply here:

  • If Claude Code does not read a matching file, the rules file is never loaded.
  • If the rules content is loaded early in the session and then buried by subsequent messages, it may be effectively ignored.

One thing to watch out for: **do not place path-scoped rules in subdirectories of .claude/rules/ and expect them to work based on their location. The subdirectory organization is cosmetic, what determines loading is the paths: frontmatter, not where the file sits within rules/. Also, user-level rules at ~/.claude/rules/ with paths: frontmatter only match files that are children of the working directory Claude Code was launched from. They will not scope correctly across arbitrary repositories.

3. The Cowboy Way: Semi-Manual Read Enforcement

Both of the above approaches share the same fundamental weakness: they depend on Claude Code deciding to read a file before the relevant instructions enter context. Sometimes that does not happen. Sometimes the instructions land in the wrong part of context. Sometimes Claude Code is tasked with something completely new and never reads anything.

If you want real control, take matters into your own hands.

  • 1. Create an agent_docs/ directory somewhere in your repo. Keeping it inside .claude/ is clean and keeps project tooling consolidated: 

    
          project_root/
          ├── .claude/
          │   ├── agent_docs/
          │   │   ├── backend-conventions.md
          │   │   ├── frontend-conventions.md
          │   │   ├── testing-patterns.md
          │   │   └── data-pipeline-guide.md

    The name agent_docs is just a convention, call it whatever you like. The location does not matter either.

  • 2. Populate it with focused, task-specific instruction files. Each file covers one domain or concern.

  • 3. In your root CLAUDE.md, add a section that describes these files and tells Claude Code when to consult each one:

    FILE · CLAUDE.md 
    
              ## Additional Reference Files

              When working on tasks in the following areas, read the corresponding
              file from .claude/agent_docs/ before proceeding:

              - Backend API development → .claude/agent_docs/backend-conventions.md
              - React components → .claude/agent_docs/frontend-conventions.md
              - Writing or editing tests → .claude/agent_docs/testing-patterns.md
              - Data ingestion or transformation → .claude/agent_docs/data-pipeline-guide.md

  • 4. Tell Claude Code, in that same section, to make the call autonomously:

    FILE · CLAUDE.md 
    
              ## Additional Reference Files

              Use your judgment to determine which files are relevant. Read them
              autonomously before starting work on the corresponding area.

How it plays out

With this setup, Claude Code may decide to read the relevant file on its own when it understands the task. You have given it explicit guidance on what exists and when to use it. That is your fallback layer.

But here is where the "cowboy" part comes in: you do not rely on that fallback. Whenever you know which agent_docs file is relevant to a task, you @-mention it in your message:

CLI 

          @.claude/agent_docs/frontend-conventions.md
          Refactor the UserProfile component to use the new design token system.

Claude Code will read that file as part of processing your message. Its content lands in the most recent part of the context window, exactly where it needs to be. No burial. No missed loads. No ambiguity.

This approach has a layered quality that makes it robust:

  • Primary layer: You @-mention the file when you know it is needed.
  • Fallback layer: Claude Code may autonomously read it based on your CLAUDE.md instructions, even if you forget.

You have full authorship over what these files contain and complete confidence over when they are read. The tradeoff is that it requires slightly more discipline in how you write your prompts. In practice, once the habit of @-mentioning the relevant doc is established, it becomes second nature.

Choosing an Approach

These three methods are not mutually exclusive. A real project might use all of them:

  • Nested CLAUDE.md for per-module conventions that naturally align with directory structure
  • Path-scoped rules for cross-cutting concerns that apply to specific file types across multiple directories (e.g., all TypeScript files, all test files)
  • agent_docs with manual @-mentions for heavyweight reference guides, onboarding documentation, or context that should only be loaded for specific high-level tasks.

The hierarchical and rules approaches give you passive, automatic loading that works well when Claude Code naturally reads the relevant files. The agent_docs approach gives you active, deliberate loading that works in all cases. Use the combination that fits your project's shape and your own working style.

The goal is always the same: when Claude Code is working on a specific task, its context should include exactly what it needs and nothing it does not.
