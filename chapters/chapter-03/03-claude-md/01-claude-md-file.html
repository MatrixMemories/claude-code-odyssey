The CLAUDE.md file

What It Is, Why It Matters, and How to Build It Right

CLAUDE.md is the single most powerful way to turn Claude Code from a forgetful blank slate into a true extension of your brain.

Every time you start a new session, Claude knows absolutely nothing about your codebase, until you give it the right context. This guide shows exactly how to create a well-structured CLAUDE.md (plus its global and local siblings) that instantly teaches the model your tech stack, your philosophy, and your exact workflow.

Subsection 1/8 ~5 min read Claude Code ≥ 2.1

LLMs are stateless functions. After their training concludes and by the time they are deployed for wide scale inference, their weights get frozen. This implies that any LLM, including the one that will run your Claude Code Agent, does not learn over time. This means that the only information known to the LLM about your codebase are the tokens supplied during one session.

Similarly, the Agentic Harness itself, being a pure facilitator between your codebase and the LLM, does not learn or remember over time neither. Therefore, we have the situation where any Coding Agent, including Claude Code, does not know absulutely anything about your codebase at the beginning of each session. Thus, you need to provide all relevant information about your codebase to the Coding Agent each time you start a session.

The CLAUDE.md file is the preferred way of providing that information to Claude Code. The CLAUDE.md file is a plain Markdown file (.md stands for Markdown). It sits in your project root directory and as per Anthropic, it is loaded at the start of every conversation and forwarded, by the harness to the LLM as context. Note that the CLAUDE.md naming is case-sensitive and the location of the file is extremely important. If you are working on a project whose root directory is project_root, your project structure should look somewhat similar to the following: 


      ~/project_root/
      ├── src/
      ├── tests/
      ├── data/
      ├── static/
      ├── CLAUDE.md                 ← Exact name, case-sensitive, in project root
      ├── pyproject.toml
      ├── README.md
      └── requirements.txt

Note that there is no need to manually mention @ your CLAUDE.md file. Once you add the CLAUDE.md file to your project_root directory, it will onboard Claude Code to the full codebase sitting in the project_root directory every time a new session is started.

CLAUDE.md Structure — WHAT / WHY / HOW

A well-structured CLAUDE.md file is organized around three questions — what the project is, why it is built the way it is, and how Claude Code must behave while working on it.

  • WHAT: the raw facts and map of your project (tech stack, folders, structure).

    This section supplies Claude Code with an immediate, objective blueprint of the project. It gives exact technology stack, directory layout, and functional scope. Only verifiable details are included: no opinions, no rationale, no preferences. The goal is to equip the model with a complete mental model of the codebase from the very first interaction, equivalent to handing a senior engineer the definitive inventory of tools and a precise floor plan of the workspace.

    Example information to include:

    • The project uses [LANGUAGE] [VERSION], the [PRIMARY FRAMEWORK/LIBRARY] [VERSION], and supporting libraries such as [LIB1], [LIB2], and [LIB3] for [core tasks].
    • The main folders are: src/ for all source code and the entry point, tests/ for unit and integration tests, docs/ for documentation and assets, assets/ for static resources (icons, samples, config files), and scripts/ for automation utilities.
    • The project is a [TYPE OF APPLICATION, e.g. desktop app / web service / CLI tool / library] that [high-level purpose in one sentence].

    This ensures Claude Code never guesses about the environment or dependencies and therefore never produces incompatible or non-functional code.

  • WHY: the purpose and reasoning behind every choice (why this project exists, why these tools and architecture).

    This section communicates the project's mission and the deliberate strategic decisions behind every major technology and design choice. By understanding the “why,” Claude Code can evaluate trade-offs, prioritize features, and generate solutions that advance the intended outcome rather than defaulting to generic or fashionable alternatives. It is the equivalent of a concise executive briefing that aligns the entire team on vision, constraints, and non-negotiables.

    Example information to include:

    • The purpose of the project is to [core mission in one clear sentence, e.g. “build a fast, private, offline-first tool for X”].
    • We chose [PRIMARY FRAMEWORK/LIBRARY] because it provides [key advantage 1] and [key advantage 2] while meeting our requirements for [performance / security / maintainability].
    • We selected [LANGUAGE] for its [strength that matches the project, e.g. speed / safety / ecosystem].
    • We deliberately avoid [ALTERNATIVE FRAMEWORK/TECHNOLOGY] because it would introduce [specific drawback that conflicts with goals, e.g. cloud dependency / excessive complexity / licensing issues].

    This alignment prevents Claude Code from repeatedly suggesting framework swaps, cloud migrations, or architectural changes that contradict the project's core constraints and goals.

  • HOW: the exact behaviors, workflows, and rules Claude must follow.

    This section defines the precise operational rules, development workflows, and quality standards that Claude Code is required to observe at all times. It translates your personal habits and best practices into actionable directives and covers commands, testing sequences, formatting, error handling, and prohibited patterns. This enables the model to work like an extension of your own workflow, rather than a generic assistant.

    Example information to include:

    • Always work inside the project’s virtual environment (created with the standard command for your language).
    • Launch the application or run tests exclusively with the documented entry-point command (never direct file execution).
    • Before implementing any new feature, first add or update the relevant test in the tests/ folder, then implement the code, and finally run the full test suite.
    • Never use debug-only statements (print, console.log, etc.) in production code, use the project’s configured logging system instead.
    • After every change, run the formatting and linting commands in this exact order: [formatter command], [import sorter command], and [linter/type checker command].
    • On any error or unexpected behavior, first add a failing test that reproduces the issue, then fix the code while preserving the test.
    • All API or external service calls must include retry logic with exponential backoff and full context logging (inputs, parameters, and stack trace).

    This directs Claude Code to follow your exact development cadence, tooling preferences, and quality bar on every single contribution.

⚠ WARNING
The CLAUDE.md is treated as context by the LLM rather than hard enforcement rules. This means, certain parts, or in the worst case the whole content, of the CLAUDE.md might be ignored by the model. It is widely speculated that the Anthropic models are trained to ignore bad instructions in the CLAUDE.md files and thus, certain instructions get ignored if they are deemed "bad" by the model.

Beyond the Repository level CLAUDE.md file

As you may have noticed, the mentioned CLAUDE.md file is dictating only rules regarding the general setup of a specific project. It lacks any instructions that are related to your personal, project-agnostic, preferences or your local machine setup. To address this shortcoming, Claude Code supports two additional variants of the CLAUDE.md file.

CLAUDE.md.local
  • Local File
  • Project Specific
  • Never committed
  • Machine-specific overrides
CLAUDE.md
  • (Project) Root File
  • Project Specific
  • Always committed
  • Project-specific instructions
~/.claude/CLAUDE.md
  • Global File
  • Project Agnostic
  • Never committed
  • Developer-specific instructions

      ~/(your home directory)
      ├── .claude/
      │   ├── CLAUDE.md                    ← Global file
      │
      ├── projects/
      │   ├── project_root/
      │   │   ├── src/
      │   │   ├── data/
      │   │   ├── static/
      │   │   ├── CLAUDE.md                ← Project specific file
      │   │   ├── CLAUDE.local.md          ← Local file
      │   │   ├── pyproject.toml
      │   │   ├── README.md
      │   │   └── requirements.txt

Global CLAUDE.md file

First, the global file, located at ~/.claude/CLAUDE.md, is your personal, machine-wide configuration that applies across every project you work on . It is intended for project-agnostic preferences like coding style, default language choices, personal tooling shortcuts or any other instruction you would otherwise repeat in every project-level file.

It is a good idea to structure your ~/.claude/CLAUDE.md file using the same what/why/how framework.

  • WHAT: your universal technical preferences and conventions (naming conventions, formatting rules, architectural patterns).
    • Always use [PREFERRED NAMING CONVENTION] for variables, functions, classes and files.
    • All code must follow [YOUR FORMATTING STANDARD] with a maximum line length of [NUMBER] characters.
    • Favor [PREFERRED ARCHITECTURAL PATTERN] and [DATA HANDLING APPROACH] where possible.
    • Use [PREFERRED DOCUMENTATION STYLE] for all public functions and complex logic.
  • WHY: your personal coding philosophy and values.
    • I prioritize [PRIMARY VALUE, e.g. readability / maintainability / simplicity / security] over cleverness or micro-optimizations.
    • Never use [PATTERN OR TOOL YOU CONSISTENTLY REJECT] because [REASON FOR REJECTION]
    • I strongly prefer [EXPLICIT / SIMPLE / MODULAR] over implicit or complex solutions.
    • Performance or new features matter only after correctness and clarity are achieved.
  • HOW: the universal rules for how Claude should behave and communicate.
    • Never generate code without [YOUR UNIVERSAL REQUIREMENT, e.g. type annotations / docstrings / error handling]
    • When suggesting changes, always show the before/after in [PREFERRED DIFF FORMAT].
    • If you are unsure about anything, explicitly state your assumptions and ask for clarification rather than guessing.
    • Never use deprecated patterns or approaches - always suggest modern, maintainable alternatives.
    • Default to writing comprehensive documentation and comments for any complex logic.

Local CLAUDE.md file

Second, the local file, named CLAUDE.local.md, which is placed in your project root, next to the CLAUDE.md file, is designed for machine-specific or developer-specific overrides that should never be committed to version control. It is ideal for capturing environment-specific paths, credentials references, local database connection strings or personal workflow tweaks that are only relevant to your own machine and do not affect your teammates. For this reason, the CLAUDE.local.md file should never be commited to shared repositories. Ideally, it should always be added to your .gitignore.

The CLAUDE.local.md file is intended to be kept short, flat list of machine-specific overrides rather than a structured briefing. It is designed to be a patch rather than an onboarding document. Therefore, it is ideal to structure it as a flat, scannable list with one-line comments above each entry explaining why the override exists locally. It should be kept as short and as specific as possible. If you find yourself writing paragraphs in your CLAUDE.local.md file, that content almost certainly belongs somewhere else.

  • What belongs in the CLAUDE.local.md file:
    • Local path overrides: any absolute or machine-specific paths that differ from the project defaults, such as [LOCAL DATA DIRECTORY], [LOCAL MODEL WEIGHTS PATH], or [LOCAL BINARY LOCATION].
    • Environment-specific commands: alternative run or build commands that only apply to your machine, such as a different [LOCAL DEV SERVER COMMAND] or [LOCAL TEST RUNNER INVOCATION] caused by a non-standard local setup.
    • Local credentials references: not the credentials themselves, but pointers to where they live locally, such as [LOCAL ENV FILE PATH] or [LOCAL SECRETS MANAGER REFERENCE].
    • Machine-specific tooling differences: if your local machine uses a different version of [TOOL OR RUNTIME] than the project default, or if you rely on a [LOCAL PROXY OR MOCK SERVICE] during development.
    • Personal workflow overrides: short-term or personal preferences you want active only in your local sessions, such as preferring more verbose logging output or skipping a specific [SLOW LOCAL TEST SUITE] during rapid iteration.
ℹ NOTE
When multiple CLAUDE.md files are present, Claude Code merges their contents by loading them alongside each other. There is no strict precedence rule between levels; conflicting instructions depend on how Claude interprets the combined text. This layered system lets you maintain a clean separation between what is universal to you as a developer, what belongs to the project and what is strictly local to your current environment.
