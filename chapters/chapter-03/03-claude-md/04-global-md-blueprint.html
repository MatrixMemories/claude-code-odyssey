Global CLAUDE.md Blueprint

The Highest-Leverage Config File You’ll Ever Write

The global ~/.claude/CLAUDE.md is quite possibly the highest-leverage configuration file you will ever write for Claude Code.

This short, permanent blueprint encodes your personal developer identity. It includes your preferred response style, stack choices, autonomy rules, testing philosophy, code review voice, and non-negotiables. Create it once, and never repeat yourself again.

Subsection 4/8 ~12 min read Claude Code ≥ 2.1

Similar to the repository level CLAUDE.md file, the Global CLAUDE.md file is best writen in sections. However, the sections themselves should be somewhat different. Remember that in this CLAUDE.md file your personal developer identity lives. Not project-specific conventions, but your universal defaults, preferences and non-negotiables.

Even more than the repository level file, the Global CLAUDE.md file is probably the most leveraged config file you will ever write for Claude Code. You create it once and benefit on every project, forever. Below we break down the essential sections — what they do, why they matter, and how to get them right.

ℹ NOTE
Most of the same do's and don'ts apply as for the repository level CLAUDE.md file. A difference to the repository level file is that the global file should be considerably shorter. Recommendation is anywhere between 30 and 100 lines.

Again, the following sections are only examples and suggestions to present how and what can be included in the global CLAUDE.md file. To keep within the recommended limits of the size, do not include all of them, rather pick and choose the most important ones.

1. Response Style (Essential)

Claude's default communication style is calibrated for general audiences. It is verbose, heavily caveated, and explanation-heavy. Since the Global CLAUDE.md is read at the start of every session, defining your preferred tone here shapes every single interaction before you type a word.

Claude Code uses this to decide how much commentary to wrap around diffs and shell output, whether to produce terse one-liners or documented explanations, and how to structure multi-step plan outputs.

Without this, you re-prompt style preferences on every new project or conversation. Centralising it means even a brand-new repo inherits your preferences from minute zero.

file · ~/.claude/CLAUDE.md 

          ## Response Style

          - Be concise. Prefer short answers; expand only if asked.
          - No motivational preamble ("Great question!", "Sure, I'd be happy to...").
          - When showing code changes, default to diffs, not full file rewrites.
          - Use inline comments sparingly — only for non-obvious logic.
          - Bullet lists over prose for multi-part answers.
          - If you're uncertain, say so in one sentence. Don't hedge every line.
⚡ Power User Hacks
  • Use a persona anchor: "Write as a senior engineer talking to another senior engineer" — this collapses explanation length dramatically.
  • Separate output modes: Define one style for code tasks and a slightly looser style for design/architecture discussions where you want more reasoning.
  • Call out the anti-patterns you hate most explicitly — "never start a response with 'Certainly'" — Claude will avoid them reliably.

2. Stack Preferences (Essential)

Without explicit stack preferences, Claude will default to the most common tool for a given task, which may not match your ecosystem at all. Declaring your canonical choices eliminates the "which library should I use?" back-and-forth entirely.

Claude Code uses this when generating scaffolding, writing new modules, suggesting dependencies, or writing test boilerplate. It won't reach for Requests when you use HTTPX, or suggest Flask when your shop runs FastAPI.

Every new project Claude touches inherits your defaults, so you spend zero time course-correcting early suggestions. This is especially high-value for AI/ML work where the ecosystem is fragmented (PyTorch vs JAX, Pandas vs Polars, etc.).

file · ~/.claude/CLAUDE.md 

          ## Stack Preferences

          ### Python / Backend
          - Python 3.12+. Type hints on all public functions.
          - Async: asyncio + anyio. Never threading for I/O.
          - HTTP client: httpx (not requests). HTTP server: FastAPI.
          - Data: Polars preferred over Pandas for new code.
          - ML: PyTorch + Lightning. Experiment tracking: MLflow.
          - Config: Pydantic BaseSettings + .env files.
          - Packaging: uv. No setup.py, no pip install -e.

          ### Frontend
          - React 18 + TypeScript. Tailwind for styling.
          - State: Zustand (not Redux). Server state: TanStack Query.
          - Never suggest class components or legacy context API.
⚡ Power User Hacks
  • "Never suggest X" statements are more powerful than positive preferences alone — Claude will actively avoid banned libraries.
  • Add a version note for things that changed significantly (e.g., "PyTorch 2.x compile API, not 1.x-style hooks") to avoid outdated patterns.

3. Define Your "Never Ask Me Again" List (Essential)

Claude's default behavior is to confirm, clarify and hedge before acting. For experienced users, this is friction. This section tells Claude exactly which decisions it has standing permission to make without asking, saving dozens of unnecessary confirmation rounds per week.

In agentic tasks, including multi-step code edits, file creation, running commands, Claude Code otherwise pauses repeatedly for micro-approvals. This list converts those pause points into automatic continuations. It's the difference between an assistant and an agent.

These preferences are personal and project-independent. You'll always want Claude to format on save and always want it to skip "should I create this directory?". There's no reason to re-declare this in every repo.

file · ~/.claude/CLAUDE.md 

          ## Never Ask Me

          - Whether to create a directory if the path is implied by the task.
          - Whether to add __init__.py files in Python packages.
          - Whether to run ruff or black after generating Python code.
          - Whether to write tests when I say "implement X" — always write them.
          - Whether to use my preferred stack (see Stack Preferences).
          - Whether to overwrite a file if I explicitly told you to rewrite it.

          ## Always Ask Me

          - Before deleting files or data.
          - Before running database migrations.
          - Before committing or pushing to git.
⚡ Power User Hacks
  • Pair with an "Always Ask Me" counterlist — the contrast makes the boundary explicit and prevents Claude from over-applying the autonomy grant to genuinely risky operations.
  • Grow this list organically: every time you find yourself saying "yes, go ahead" to the same question, add it.

4. Testing Philosophy (Essential)

Testing preferences vary wildly between developers and are almost never reflected in a project's boilerplate. Without guidance, Claude will produce tests that are technically valid, but misaligned with how you actually verify correctness. This will reflect in wrong granularity, wrong framework, or wrong coverage focus.

Claude Code generates test stubs alongside implementation code. This section determines the framework, the assertion style, the mocking approach, and how deeply Claude tests edge cases vs. just the happy path.

Test strategy is usually one of the most personal engineering choices and one of the least likely to change between projects. Encoding it globally means every implementation Claude writes, will come with tests that actually fit your workflow.

file · ~/.claude/CLAUDE.md 

          ## Testing Philosophy

          - Framework: pytest. No unittest.TestCase subclasses.
          - Fixtures over setUp/tearDown. Use conftest.py.
          - Test behaviour, not implementation. No testing private methods.
          - Mocking: pytest-mock (mocker fixture). Prefer dependency injection
            over patching when possible.
          - Coverage goal: 80% on business logic, 0% on migrations/scripts.
          - Integration tests go in tests/integration/, unit in tests/unit/.
          - ML models: always test with a minimal synthetic dataset, not prod data.
          - Parametrize edge cases. Don't write five near-identical test functions.
⚡ Power User Hacks
  • Add a naming convention rule (e.g., "test_{thing}_{condition}_{expected_result}"). Claude will follow it consistently and your test suite becomes self-documenting.
  • If you do Test-Driven Development (TDD), say it explicitly: "Write the test first, then the implementation." Claude will invert its default generation order.

5. Encode Your Code Review Voice (High Value)

When you ask Claude to review your code, it defaults to a diplomatic, comprehensive, enumerate-everything style. Your actual review preferences — what to nitpick, what to ignore, how to phrase criticism — are entirely personal and take months for a human colleague to learn.

Claude Code uses this section to calibrate the severity, focus, and format of review feedback. The difference between "provide feedback" and a well-tuned review voice is the difference between generic linting commentary and actionable senior-engineer input.

Review preferences don't change between projects. Whether you're reviewing a PyTorch training loop or a React component, you want the same signal-to-noise ratio and the same level of directness.

file · ~/.claude/CLAUDE.md 

          ## Code Review Voice

          - Be direct. Label issues: [blocking], [suggestion], [nit].
          - Don't praise correct code. Focus on what needs to change.
          - Blocking issues: correctness bugs, security holes, perf cliffs.
          - Suggestions: better patterns, API improvements, readability.
          - Nits: style, naming — mention once, don't dwell.
          - Always suggest the fix, not just the problem.
          - Skip comments on auto-generated code or vendored files.
          - Max 10 comments per review. Prioritise ruthlessly.

6. Git & Commit Convention (High Value)

Commit history is a communication medium, it documents intent, enables bisecting, and drives changelogs. Without explicit convention, Claude will invent its own style on every git commit it touches, producing an inconsistent log that's noise rather than signal.

Claude Code generates commit messages when you ask it to stage and commit changes, writes PR descriptions, and summarizes diffs. This section controls every word of that output, including type prefix, tense, scope format, body length, and whether breaking changes get flagged. It also shapes how Claude reads your history when it needs to reason about what changed and why.

Commit style is personal and cross-project. Whether you're on a solo ML experiment or a shared API service, you want the same readable history. Declaring it globally means Claude never reverts to generic "fix bug" messages, even on a repo it has never seen before.

The following types form the full vocabulary Claude will draw from when writing commit messages. Scope and subject rules apply to all of them equally:

Type When to Use Example
feat New feature, new capability feat(auth): add refresh token rotation
fix Bug fix fix(trainer): prevent NaN loss on empty batch
refactor Code change with no behaviour change refactor(pipeline): extract transform step
chore Tooling, deps, CI, config chore: bump torch to 2.3.0
docs Documentation only docs(api): document pagination params
test Adding or updating tests test(model): add edge case for zero-length input
perf Performance improvement perf(dataloader): switch to pin_memory prefetch
file · ~/.claude/CLAUDE.md 

          ## Git & Commit Convention

          ### Commit message format
          Follow Conventional Commits: <type>(<scope>): <subject>
          - Subject: imperative mood, lowercase, no period, max 72 chars.
            Good: "add retry logic for transient errors"
            Bad:  "Added retry logic." / "Adds retry logic"
          - Scope: the module, file, or subsystem affected (optional but preferred).
          - Breaking changes: append ! to type (feat!) AND add footer:
            BREAKING CHANGE: <what breaks and migration path>

          ### Commit body (include when the why isn't obvious)
          - Blank line after subject, then wrap at 72 chars.
          - Explain WHY, not WHAT. The diff shows what changed.
          - Reference issues: Closes #123 or Refs #456 in footer.
          - Skip the body for chore/docs/test unless context is critical.

          ### PR / MR descriptions
          Structure: ## What, ## Why, ## Testing, ## Notes (optional)
          - Keep "What" to 3 bullet points max.
          - "Testing" must mention how to verify the change locally.
          - Tag any follow-up work as ## TODO with issue links.

          ### Branch naming
          <type>/<short-slug>  →  feat/refresh-token-rotation
                                  fix/nan-loss-empty-batch
          No ticket numbers in branch names unless team policy requires it.

          ### What NOT to commit
          - Never commit: .env files, API keys, model weights, __pycache__.
          - Never squash mid-PR without asking — I may want the history.
          - Never amend commits that have been pushed to a shared branch.
⚡ Power User Hacks
  • Add a changelog generation instruction: "When I ask for a changelog, group commits by type and write it in keep-a-changelog format." Claude will produce a release-ready CHANGELOG.md from your history automatically.
  • Teach it your scope vocabulary: List the actual module names in your codebase (e.g. trainer, api, ingest, ui). Claude will pick the right scope instead of inventing generic ones like core.
  • Define a "commit this work" shorthand: In your Never Ask section, add "When I say 'commit this', write the message, show it to me, then wait for approval." This creates a fast review loop without losing control of the history.
  • Pair with your CI rules: If your pipeline rejects commits without a type prefix (common with commitlint), state it here. Claude will never write a bare message again, even in a repo with no commitlint config present yet.

7. Specify Your Error Message Standard (High Value)

Error messages are user-facing contracts. Inconsistent error design — mixing exception types, formats, and detail levels — creates debugging friction and poor user experience. Claude will copy whatever pattern it sees in context without guidance.

When Claude writes exception handling, validation logic, or API error responses, it uses this to choose exception hierarchy, message verbosity, and whether to include context variables. Consistent error design is only achievable at the global level if Claude is generating code across multiple projects.

Error standards span your entire codebase ecosystem. Declaring them globally means every new service or module Claude helps build speaks the same error language, reducing cognitive overhead when debugging across services.

file · ~/.claude/CLAUDE.md 

          ## Error Message Standard

          - Raise specific exceptions, never bare Exception.
          - Error messages: "{what failed}: {why} (got: {actual_value})"
          - Always use `raise X from e` to preserve the chain.
          - API errors: return {"error": {"code": "...", "message": "...", "details": {...}}} — never expose raw stack traces.
          - Log at ERROR for unexpected exceptions, WARNING for handled ones.
          - Include the offending value in the message when safe to do so.

8. Teach It Your Debugging Ritual (High Value)

Debugging is deeply personal, some developers want hypotheses before code, others want immediate reproductions, others want structured root-cause analysis. Claude defaults to a generic "let me check a few things" approach that rarely matches how you actually debug. This section also replaces the "Scratchpad Convention" section, since at global scope a scratchpad is simply Claude's internal debugging workspace format.

When you paste a stack trace or describe a bug, this shapes whether Claude jumps to code, asks for reproduction steps, or writes a structured hypothesis tree. It also governs how Claude uses its own "thinking space", whether it reasons out loud or presents a clean conclusion.

Your debugging instincts are consistent across tech stacks. The five-why habit you use for a Python ML bug is the same one you apply to a React rendering issue. Encoding it globally makes Claude a natural extension of how you think through problems.

file · ~/.claude/CLAUDE.md 

          ## Debugging Ritual

          When I share a bug, follow this order:
          1. State the most likely cause in one sentence.
          2. List 2–3 alternative hypotheses briefly.
          3. Provide the minimal reproduction or diagnostic snippet.
          4. Only then suggest the fix.

          If you need to reason through a complex problem, use a
          <scratchpad> block — I'll read it, but it won't clutter the answer.

          Never jump straight to "try this fix" without a stated hypothesis.
          For ML bugs: always check data shape, dtype, and device first.
⚡ Power User Hacks
  • The <scratchpad> convention is powerful: it lets Claude reason at length without polluting your terminal output. Define the tag name here so it's consistent.
  • Add domain-specific heuristics: "For CUDA errors, always check device placement before anything else". Claude will front-load the highest-signal checks for your stack.

9. Set a Complexity Budget (Optional)

Claude is biased toward thorough, complete solutions which often means over-engineered ones. A complexity budget is a standing instruction to prefer the boring solution, resist abstraction before it's earned, and flag when a request is heading toward unnecessary complexity.

Without this, Claude will reach for design patterns, abstract base classes, and plug-in architectures for problems that a 20-line function would solve.

Complexity creep is a universal anti-pattern, not a per-project concern. A global budget ensures Claude's default solution instinct stays calibrated to your taste, regardless of which codebase it's in.

file · ~/.claude/CLAUDE.md 

          ## Complexity Budget

          - Default to the simplest thing that could possibly work.
          - Prefer functions over classes until state is clearly needed.
          - No premature abstraction. Abstract after the third repetition.
          - If a solution requires more than 2 new files, flag it and
            ask if I want the simple or full version.
          - Avoid introducing new dependencies for things stdlib can do.
          - Prototypes and scripts: optimise for readability, not architecture.
⚡ Power User Hacks
  • Define a "complexity escalation trigger": "If your solution would take more than 30 minutes to explain, stop and propose a simpler alternative first." This keeps Claude from going deep into an over-engineered path silently.
  • You can override this per-project: a production ML pipeline CLAUDE.md can explicitly say "full architecture expected here."

10. Prime It With Your Blind Spots (Optional)

Every developer has recurring mistakes, patterns they reach for incorrectly, edge cases they forget to handle, abstractions they over-use. Encoding these as standing warnings turns Claude into a personalised second reviewer that specifically knows your failure modes.

Claude will proactively check for and flag the patterns you've listed, even when you don't ask. It transforms passive code generation into active quality control tailored to you, not some generic checklist.

Your blind spots follow you across every project and language. A global reminder ensures you get warned about your specific patterns in a Python repo, a TypeScript component, and a bash script alike.

file · ~/.claude/CLAUDE.md 

          ## My Blind Spots — Proactively Flag These

          - I tend to skip input validation on internal functions. Always add it.
          - I forget to handle the empty list / None case in transforms.
          - I reach for global state when a passed parameter would do.
          - I underestimate GPU memory. Flag any operation that could OOM.
          - I write async code but forget to await it. Call this out explicitly.
          - I deploy without checking that env vars are documented in .env.example.
⚡ Power User Hacks
  • This list is most powerful when it's honest and specific. Vague entries ("I sometimes write bad code") do nothing. Specific patterns ("I often mutate the input list instead of copying it") get actionable warnings.
  • Review and update this section every few months. Blind spots evolve as you grow. When Claude's warnings stop feeling surprising, you've internalised that lesson, remove it and add new ones.

11. Add a "What Good Looks Like" Benchmark (Optional)

Positive examples are the fastest way to align Claude's quality bar with yours. Rather than describing what to avoid, you show the finish line and Claude aims for it by default, even without explicit instruction.

Claude uses the benchmark as an implicit target for code it generates. A well-chosen example calibrates naming conventions, docstring depth, function length, abstraction level, and error handling in one shot — more effectively than prose rules alone.

Quality standards transfer across domains. The same taste that makes a Python function excellent, informs TypeScript, Bash, and SQL output. A global benchmark creates a unified aesthetic across everything Claude produces for you.

file · ~/.claude/CLAUDE.md 

          ## What Good Looks Like

          A function I'd be proud to ship:
          - Does one thing. Fits on a screen without scrolling.
          - Has a one-line docstring stating what it returns, not what it does.
          - Raises a descriptive exception for invalid input; doesn't silently return None.
          - Has a single test that covers the most important edge case.
          - Uses the type system to prevent the bug, not a runtime check.

          A PR I'd be proud to open:
          - Single, atomic purpose. Could be reverted without side effects.
          - Passes CI without my intervention.
          - Has a description that explains the *why*, not the *what*.
⚡ Power User Hacks
  • Define benchmarks at multiple scales: a good function, a good module, a good PR. Claude will calibrate to the appropriate scope depending on the task.
